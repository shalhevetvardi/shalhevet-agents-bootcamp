# Knowledge Management Guide — ניהול מסמכי ידע

> מסמך קבוע. מסביר לי (Agent 2) איך לנהל את מסמכי הידע של המשתמש שלי.
> מסמך תפעולי — לא ידע על המשתמש, אלא הוראות עבודה.

---

## ארכיטקטורה: Master Library + סוכנים

```
[שם-הריפו]/
├── Master Library/                          ← מקור האמת היחיד
│   ├── DOCUMENT-MAP.md                      ← מפת המסמכים (אני מתחזק)
│   ├── .knowledge-sync/
│   │   └── manifest.yaml                    ← רישום מובנה (אני מתחזק)
│   ├── knowledge-management-guide.md        ← אני (ההוראות שלי)
│   ├── knowledge-about-me.md
│   ├── technical-context.md
│   ├── tools-and-environment.md
│   ├── ... (כל מסמכי הידע)
│
├── סוכנים/
│   ├── בפיתוח/
│   │   └── [שם-סוכן]/
│   │       ├── system-prompt.md
│   │       └── Master Library/              ← עותק (מסונכרן מהשורש)
│   └── מוכנים/
│       └── [שם-סוכן]/
│           ├── system-prompt.md
│           └── Master Library/              ← עותק (מסונכרן מהשורש)
```

### עיקרון מפתח: מקור אמת אחד

- **`Master Library/` בשורש** = מקור האמת. כל עריכה קורית שם.
- **עותקים בתיקיות סוכנים** = מועתקים מהשורש. לא עורכים ישירות בעותק.
- כשמשהו משתנה בשורש → צריך סנכרון לכל הסוכנים שיש להם עותק.

## שני התוצרים שאני מתחזק

מעבר למסמכי הידע עצמם, אני אחראי על שני קבצי מטא:

### 1. `DOCUMENT-MAP.md` — האינדקס הנרטיבי

מסמך קריא לבני אדם. מסכם: אילו מסמכים יש, מה כל אחד מכיל, מתי להשתמש בו, ומה הקשרים בינו למסמכים אחרים.

**מבנה:**
```markdown
# DOCUMENT MAP — [שם המשתמש] / [שם הריפו]

> מטרה: לאפשר ל-Agent 2 (ולסוכנים נוספים) לגשת למידע הנכון במהירות.

## קטגוריות

| קטגוריה | מספר מסמכים | פוקוס |
|---------|-------------|-------|
| 🧑 ידע אישי | X | מי המשתמש, העסק, הכלים, הרמה הטכנית |
| 🎓 ידע קורס | X | חומרי הקורס, שיטות, פרויקטים |
| ⚙️ ידע תפעולי | X | הוראות עבודה, ארכיטקטורה |

**סה"כ:** [N] מסמכים.

---

## אינדקס מפורט

### 🧑 ידע אישי

#### Document 1: knowledge-about-me.md
- **Path:** `Master Library/knowledge-about-me.md`
- **Language:** עברית
- **Contains:** [תיאור קצר של מה המסמך מכיל]
- **Use when:** [מתי להשתמש בו]
- **Key topics:** [נושאי מפתח]
- **Related:** [מסמכים קשורים]

[המשך לכל מסמך...]
```

### 2. `.knowledge-sync/manifest.yaml` — הרישום המובנה

קובץ JSON/YAML למכונה. מכיל את כל מה שצריך כדי לסנכרן: hashes, sync targets, קטגוריות.

**מבנה (פשוט מהגרסה של שלהבת):**
```json
{
  "generated_at": "[תאריך]",
  "policy": {
    "library_is_canonical_for_shared": true,
    "silent_overwrite_allowed": false
  },
  "documents": [
    {
      "name": "knowledge-about-me.md",
      "doc_id": "doc-[uuid או hash קצר]",
      "canonical_path": "Master Library/knowledge-about-me.md",
      "category": "shared",
      "sync_targets": ["[שם-סוכן-1]", "[שם-סוכן-2]"],
      "last_synced_hash": "[hash של התוכן]",
      "last_synced_at": "[תאריך]",
      "notes": ""
    }
  ]
}
```

**קטגוריות:**
- `shared` — מסונכרן לכל הסוכנים
- `agent-only` — שייך רק לסוכן ספציפי
- `library-only` — בספרייה בלבד, לא מועתק לאף סוכן

## הטקס הראשון: מיפוי ראשוני

**זה הדבר הראשון שאני עושה כשהמשתמש מפעיל אותי לראשונה.**

הטקס קורה כשהמשתמש אומר משהו כמו: "תמפה לי את הידע", "התחל מיפוי", או כשבְּנָיָה הנחה אותו לבקש את זה במפגש 8.

### הזרימה — שלב-שלב

**שלב 1: הצהרה**
> "מעולה — בוא נמפה ביחד את כל הידע שלך. זה ייקח כ-5 דקות, ובסוף יהיו לנו שני קבצים שיעזרו לנו לעבוד ביחד לתמיד: `DOCUMENT-MAP.md` (לקריאה) ו-`manifest.yaml` (לסנכרון). אסביר תוך כדי."

**שלב 2: סקירת ה-Master Library**
- אני סורק את `Master Library/`
- מציג למשתמש: "מצאתי [N] מסמכים. הנה הם:"
- רשימה קצרה — שם + שורה אחת על כל מסמך

**שלב 3: סקירת תיקיות סוכנים**
- אני סורק את `סוכנים/בפיתוח/*/Master Library/` ו-`סוכנים/מוכנים/*/Master Library/`
- מציג: "מצאתי [N] סוכנים: [רשימה]. כל אחד מהם מכיל עותק של המסמכים."

**שלב 4: בניית DOCUMENT-MAP.md**
- אני קורא כל מסמך
- מסווג לקטגוריה (אישי / קורס / תפעולי)
- כותב סעיף לכל מסמך: שם, תיאור, מתי להשתמש, נושאי מפתח
- מציג טיוטה למשתמש: "ככה ה-MAP נראה. נראה לך טוב?"
- אחרי אישור — שומר ב-`Master Library/DOCUMENT-MAP.md`

**שלב 5: בניית manifest.yaml**
- אני יוצר רישום לכל מסמך
- לכל מסמך: name, doc_id, canonical_path, category, sync_targets, hash
- שומר ב-`Master Library/.knowledge-sync/manifest.yaml`

**שלב 6: סנכרון ראשוני**
- אני בודק שכל סוכן מקבל את המסמכים שצריך
- מסכם: "[N] סוכנים סונכרנו. כל סוכן מכיל עותק עדכני."

**שלב 7: הסבר על העתיד**
> "מעכשיו, כשתרצה לעדכן מסמך — תגיד לי. אני אדאג ל:"
> "- לקרוא את הגרסה הנוכחית"
> "- להציע לך שינוי"
> "- לעדכן ב-`Master Library` (שורש)"
> "- לסנכרן לכל הסוכנים"
> "- לעדכן את ה-MAP וה-manifest"
>
> "בקיצור — אני שומר על סדר. אתה רק אומר לי מה השתנה."

**שלב 8: הצעת push**
> "אם תרצה, אנחנו יכולים עכשיו לעשות commit + push כדי לשמור הכל ב-GitHub."

## 5 פעולות שאני יודע לעשות (אחרי המיפוי הראשוני)

### 1. הצגת מצאי — "מה יש לי?"

כשהמשתמש שואל "אילו מסמכים יש לי" / "מה הידע שלי" / "תראה לי הכל":

- אני קורא את `DOCUMENT-MAP.md` ומציג סיכום
- אם המשתמש רוצה פרטים על מסמך ספציפי — אני נכנס לעומק

### 2. עדכון מסמך קיים — "תעדכן את..."

כשהמשתמש רוצה לעדכן מסמך:

1. **שואל מה השתנה** — "מה להוסיף/לשנות/להסיר?"
2. **קורא את המסמך הנוכחי** — מבין מה יש
3. **מציע עדכון** — מראה את השינוי המוצע
4. **מחכה לאישור** — "נראה לך טוב?"
5. **שומר** — מעדכן את הקובץ ב-Master Library (שורש)
6. **בודק cross-reference** — האם השינוי משפיע על מסמכים אחרים?
   - דוגמה: "שינית את תחום העסק ב-`knowledge-about-me` — רוצה שאעדכן גם את `goals-and-use-cases`?"
7. **מעדכן manifest** — מחשב hash חדש, מעדכן `last_synced_at`
8. **מעדכן DOCUMENT-MAP** — אם התיאור או הנושאים השתנו
9. **מציע סנכרון** — "המסמך עודכן בשורש. רוצה שאסנכרן לסוכנים?"

### 3. הוספת מסמך חדש — "תוסיף מסמך ידע"

כשהמשתמש רוצה ליצור מסמך חדש:

1. **מברר מה התוכן** — "על מה המסמך? מה הוא אמור לכסות?"
2. **מציע שם** — באנגלית, קצר, תיאורי (לפי המוסכמה: `kebab-case.md`)
3. **מציע מבנה** — header עם תיאור + סקשנים מתאימים
4. **כותב טיוטה** — מבוסס על דאטה שהמשתמש נתן
5. **שואל על קטגוריה** — shared / agent-only? אילו סוכנים יקבלו אותו?
6. **שומר** — ב-`Master Library/` (שורש)
7. **מוסיף ל-manifest.yaml** — רישום חדש
8. **מוסיף ל-DOCUMENT-MAP.md** — סעיף חדש בקטגוריה המתאימה
9. **מסנכרן** — מעתיק לסוכנים שצריך

### 4. סנכרון — "תסנכרן" / "sync"

כשהמשתמש מבקש סנכרון, או כשאני מזהה שעדכנתי משהו בשורש:

1. **קורא manifest.yaml** — לראות מה אמור להיות איפה
2. **סורק** — Master Library בשורש vs. עותקים בתיקיות הסוכנים
3. **משווה** — קובץ-קובץ עם hashes
4. **מציג diff** — "ב-`[שם סוכן]` הגרסה ישנה. הנה מה שהשתנה: [...]"
5. **מבקש אישור** — "לעדכן?"
6. **מעתיק** — מהשורש לעותק
7. **מעדכן manifest** — `last_synced_hash`, `last_synced_at`

**כללים:**
- שורש שונה מעותק ← **השורש מנצח**. מעתיק לעותק.
- קובץ בעותק שלא בשורש ← **אזהרה**: "נמצא קובץ בתיקיית סוכן שלא נמצא ב-Master Library. מה לעשות?"
- קובץ בשורש שלא בעותק ← בודק את `sync_targets` ב-manifest. אם כן — מעתיק.

**Terminal workflow:**
```
git add .
git commit -m "sync: Master Library → [שמות סוכנים]"
git push
```

### 5. בדיקת drift / עדכניות — "תבדוק אם הכל עדכני"

כשהמשתמש שואל, או כשעברו כמה שבועות:

1. **בודק drift טכני** — מה ב-manifest.yaml לא מסתדר עם הקבצים בפועל
   - hash שונה → צריך סנכרון
   - קובץ ב-manifest שלא קיים → להציע למחוק או להוסיף בחזרה
   - קובץ קיים שלא ב-manifest → להוסיף ל-manifest
2. **בודק drift תוכני** — האם התוכן עדיין נכון?
   - שואל שאלות ממוקדות: "ב-`knowledge-about-me` כתוב שאתה עובד עם 3 לקוחות. עדיין נכון?"
3. **מציע עדכונים** — "אני חושב ש-`tools-and-environment` צריך עדכון כי הזכרת כלי חדש בשיחה קודמת"

## מתי אני מפעיל את זה מעצמי (proactive)

- **אחרי עיבוד חומר קורס** — אם עדכנתי `course-lessons.md`, אני מציע: "רוצה שאסנכרן ואעדכן את ה-MAP?"
- **כשהמשתמש מזכיר שינוי** — "עברתי לכלי חדש" → "רוצה שאעדכן את `tools-and-environment`?"
- **כשמשהו לא מסתדר** — אם אני קורא מסמך ומה שכתוב שם לא מתאים למה שהמשתמש אמר → "שים לב, ב-[מסמך] כתוב X אבל אמרת Y. נעדכן?"

---

## מה אני לא עושה בלי אישור

- לא מוחק מסמך ידע
- לא משנה שם קובץ
- לא מוחק תוכן ממסמך (רק מוסיף או מחליף — ותמיד עם diff)
- לא עושה push בלי שהמשתמש אמר "כן"
- לא עושה sync שמשנה הרבה קבצים בו-זמנית בלי לסכם מה הולך לקרות

## כשהמשתמש בונה סוכן חדש

אם המשתמש רוצה לבנות סוכן נוסף:

1. יוצר תיקייה חדשה ב-`סוכנים/בפיתוח/[שם חדש]/`
2. יוצר `system-prompt.md` חדש
3. מעתיק את **כל** ה-Master Library מהשורש לתיקיית הסוכן החדש
4. **מעדכן manifest.yaml** — מוסיף את הסוכן החדש ל-`sync_targets` של כל המסמכים השייכים
5. **מעדכן DOCUMENT-MAP.md** — מציין שיש סוכן חדש במערכת
6. מתחיל לסנכרן גם אותו מעכשיו

**עיקרון:** כל סוכן חדש מקבל ברירת מחדל את **כל** ה-Master Library. אם בעתיד יהיו מסמכים שרלוונטיים רק לסוכן מסוים — נוסיף אותם ישירות בתיקיית הסוכן (category: `agent-only`) ולא בשורש.
