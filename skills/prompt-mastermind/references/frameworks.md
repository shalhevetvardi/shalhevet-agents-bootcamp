# Frameworks — ספריית פריימוורקים

קובץ עזר המכיל את כל 26 הפריימוורקים של הסקיל. טען אותו אחרי שזיהית intent, ועיין בפריימוורקים הרלוונטיים לאותו intent.

**מבנה אחיד לכל פריימוורק:**
- ראשי תיבות
- שורה אחת (מה זה)
- מתי להשתמש / מתי לא
- מרכיבים
- Template בעברית
- דוגמה מעולם בניית סוכנים

# א. CREATE — יצירת תוכן חדש

---

## APE

**ראשי תיבות:** Action, Purpose, Expectation

**שורה אחת:** פרומפט אולטרה-מינימלי לפעולה פשוטה וחד-פעמית.

**מתי להשתמש:** משימה פשוטה, חד-פעמית, אין צורך ב-role או הקשר מורכב. רוצים מקסימום ישירות במינימום מילים.

**מתי לא להשתמש:** משימה דורשת טון/קהל/סגנון ספציפיים, או יש צורך ב-role מפורש.

**מרכיבים:**
- **Action** — מה לעשות, בפועל.
- **Purpose** — למה זה נחוץ.
- **Expectation** — איך נראית הצלחה.

**Template (עברית):**
```
[Action]: ___
[Purpose]: ___
[Expectation]: ___
```

**דוגמה — בניית prompt לחילוץ metadata מקובץ:**

```
Action: חלץ את ה-metadata מקובץ ה-markdown הזה ותחזיר אותו כ-JSON.
Purpose: כדי שאוכל לאינדקס את הקובץ בדאטאבייס.
Expectation: JSON תקין, רק המפתחות של ה-frontmatter, בלי טקסט נלווה.
```

**Components Filling Guide:**
- **Action** — טוב: פועל + אובייקט ספציפיים ("חלץ metadata מ-markdown"). גרוע: פועל עמום ("תעזור לי", "תעשה משהו עם").
- **Purpose** — טוב: הסבר תכליתי משפט אחד ("לאינדוקס בדאטאבייס"). גרוע: "זה נחמד", "רציתי". המטרה מכוונת את הפלט.
- **Expectation** — טוב: פורמט מדויק + מה לא להכליל ("JSON, רק frontmatter, בלי טקסט"). גרוע: "משהו בסדר", "פורמט סטנדרטי".

**Before → After:**
- Before: *"תעזור לי לחלץ נתונים מהקובץ שלי בפורמט טוב"* — 3 רכיבים גנריים.
- After: *"Action: חלץ title + tags + date מ-YAML frontmatter. Purpose: אינדקס Notion. Expectation: JSON עם המפתחות האלה בלבד, null למפתחות חסרים."* — כל רכיב ספציפי, פורמט חד-משמעי.

## RTF

**ראשי תיבות:** Role, Task, Format

**שורה אחת:** פרומפט פשוט מונע-מומחיות — קלוד לובש role וממלא משימה בפורמט נתון.

**מתי להשתמש:** המשימה דורשת נקודת מבט של מומחה (עו"ד, מפתח, קופירייטר). הפלט בפורמט מוכר.

**מתי לא להשתמש:** כשאין צורך ב-role (זה לא משנה מי עושה את זה), או כשההקשר חשוב יותר מהמומחיות.

**מרכיבים:**
- **Role** — מי אתה (תחום מומחיות + שנות ניסיון / ניואנס).
- **Task** — מה לעשות.
- **Format** — איך להציג.

**Template (עברית):**
```
[Role]: אתה ___
[Task]: המשימה שלך היא ___
[Format]: הצג את הפלט כ-___
```

**דוגמה — system prompt לסוכן code reviewer:**

```
Role: אתה code reviewer בכיר עם 12 שנות ניסיון ב-Python וב-TypeScript, המתמחה ב-production code של backend.
Task: סקור את הקוד שהמשתמש מדביק וזהה בעיות של: security, performance, תחזוקתיות.
Format: הצג כרשימה ממוספרת. לכל בעיה — קטגוריה, חומרה (1-5), ושורה ספציפית בקוד. בלי סיכום כללי בסוף.
```

**Components Filling Guide:**
- **Role** — טוב: מומחיות + שנות ניסיון + ניואנס מיוחד ("code reviewer בכיר, 12 שנות ניסיון ב-backend Python"). גרוע: role גנרי ("מומחה טכני", "עוזר").
- **Task** — טוב: פועל ברור + אובייקט + סוג תוצאה ("סקור וזהה בעיות של X, Y, Z"). גרוע: "בדוק את הקוד", "תן משוב".
- **Format** — טוב: מבנה מפורש + מה לכלול + מה לא ("רשימה ממוספרת, קטגוריה+חומרה+שורה, בלי סיכום"). גרוע: "פורמט ברור", "סדר טוב".

**Before → After:**
- Before: *"אתה מפתח. תבדוק לי את הקוד ותגיד מה לא בסדר."* — role רדוד, task עמום, format חסר.
- After: *"אתה senior code reviewer עם התמחות ב-production backend. סקור את הקוד ל-security/performance/maintainability issues. החזר רשימה ממוספרת עם category+severity(1-5)+line number לכל issue."* — role עם עומק, task עם קריטריונים, format קונקרטי.

## CTF

**ראשי תיבות:** Context, Task, Format

**שורה אחת:** פרומפט פשוט מונע-הקשר — המצב חשוב יותר מהמומחיות.

**מתי להשתמש:** המצב/הסיטואציה הוא מה שמגדיר את הפלט, לא role ספציפי. לדוגמה: "אני בפגישה עם לקוח כועס" — לא משנה מי אני, משנה מה המצב.

**מתי לא להשתמש:** כשהמומחיות של המבצע היא קריטית — אז RTF.

**מרכיבים:**
- **Context** — מה קורה, מה הרקע.
- **Task** — מה לעשות.
- **Format** — איך להציג.

**Template (עברית):**
```
[Context]: ___
[Task]: ___
[Format]: ___
```

**דוגמה — סוכן שעוזר לחלץ action items משיחה:**

```
Context: לקחתי חלק בשיחת צוות של 45 דקות. המשתתפים דיברו על 3 פרויקטים במקביל וקפצו נושא-לנושא. אני צריכה לזהות מה צריך להיעשות, על ידי מי, ועד מתי.
Task: עבור על התמלול שאדביק. חלץ כל action item שהוזכר (גם אם רק רמוזה התחייבות), שייך אותו לאדם אם שמו הוזכר, ותאריך יעד אם הוזכר.
Format: טבלה עם 3 עמודות: פעולה | אחראי | דדליין. אם אין אחראי/דדליין — "לא צוין".
```

**Components Filling Guide:**
- **Context** — טוב: רקע עם specifics שמשפיעים על הפלט ("שיחה של 45 דקות, 3 פרויקטים במקביל, קפיצות"). גרוע: הקשר כללי ("יש לי שיחה").
- **Task** — טוב: פועל מדויק + מה לחפש + מה לעשות עם מה שלא ברור ("חלץ כולל רמוזות, שייך אם יש שם, כתוב 'לא צוין' אם חסר"). גרוע: "מצא משימות".
- **Format** — טוב: מבנה מפורש עם טיפול במקרי קצה ("טבלה, 'לא צוין' כשחסר"). גרוע: "רשימה נחמדה".

**Before → After:**
- Before: *"יש לי תמלול של שיחת צוות, תעזור לי למצוא מה היו ההחלטות"* — הקשר עמום, task לא מוגדר, אין פורמט.
- After: *"Context: שיחת צוות 45 דקות, 3 פרויקטים במקביל. Task: חלץ action items כולל התחייבויות רמוזות, שייך לאדם ודדליין כשאפשר. Format: טבלה 3 עמודות עם 'לא צוין' כדיפולט."* — קונטקסט מנחה, task פעולתי, טבלה מובנית.

## RACE

**ראשי תיבות:** Role, Action, Context, Expectation

**שורה אחת:** פרומפט מורכבות בינונית — משלב role + הקשר + תוצאה מפורשת.

**מתי להשתמש:** המשימה דורשת גם מומחיות וגם הקשר ספציפי, ויש תוצאה מצופה ברורה.

**מתי לא להשתמש:** כשיש כללי compliance מפורשים (אז CARE/TIDD-EC) או כשצריך multi-step (אז RISEN).

**מרכיבים:**
- **Role** — מי אתה.
- **Action** — מה לעשות.
- **Context** — באיזה מצב.
- **Expectation** — איך נראה פלט מוצלח.

**Template (עברית):**
```
[Role]: אתה ___
[Action]: המשימה שלך ___
[Context]: המצב הוא ___
[Expectation]: פלט טוב נראה כך: ___
```

**דוגמה — סוכן שכותב תגובות ל-reviews של לקוחות:**

```
Role: אתה אחראי שירות לקוחות בחברת SaaS B2B קטנה (30 עובדים), עם גישה אמפתית ופתרונית.
Action: כתוב תגובה פומבית לביקורת שלילית שהשאיר לקוח.
Context: הלקוח קיבל חיוב כפול החודש, והתלונן פומבית ב-Trustpilot. הצוות כבר זיהה את הבאג, החזיר את הכסף, ותיקן את הבעיה. הביקורת היא מלפני 3 ימים.
Expectation: תגובה של 3-4 משפטים, מתחילה בהכרה בבעיה, מציינת מה נעשה לתיקון, ומזמינה ליצירת קשר ישיר. בלי התנצלות מוגזמת, בלי הבטחות עתידיות.
```

**Components Filling Guide:**
- **Role** — טוב: תפקיד + גודל ארגון + גישה ("אחראי שירות ב-SaaS של 30 עובדים, גישה אמפתית ופתרונית"). גרוע: "שירות לקוחות טוב".
- **Action** — טוב: פועל + אובייקט + טווח (כתוב תגובה פומבית לביקורת שלילית"). גרוע: "תגיב ללקוח".
- **Context** — טוב: facts ספציפיים שמשנים את התגובה (חיוב כפול, כבר הוחזר, 3 ימים). גרוע: "לקוח כועס".
- **Expectation** — טוב: מבנה + מה לכלול + מה לא ("3-4 משפטים, מתחיל בהכרה, מסתיים בהזמנה; בלי הבטחות, בלי דרמה"). גרוע: "תגובה מקצועית".

**Before → After:**
- Before: *"לקוח התלונן על חיוב כפול, תכתוב לו תגובה יפה"* — 4 רכיבים חסרים.
- After: *"Role: CS בחברה קטנה, גישה אמפתית-פתרונית. Action: תגובה פומבית ל-Trustpilot review. Context: חיוב כפול, כבר הוחזר, הבעיה נפתרה. Expectation: 3-4 משפטים, הכרה→תיקון→הזמנה ישירה; בלי הבטחות עתידיות."* — כל רכיב מגובה ב-specifics שמשנים את התוצאה.

## CRISPE

**ראשי תיבות:** Capacity+Role, Insight, Statement, Personality, Experiment

**שורה אחת:** פרומפט יוצר מספר גרסאות פלט להשוואה.

**מתי להשתמש:** רוצים לראות מספר אפשרויות פלט שונות כדי לבחור או לשלב.

**מתי לא להשתמש:** רוצים תשובה אחת חותכת — זה יהיה overhead.

**מרכיבים:**
- **Capacity + Role** — תפקיד ויכולות.
- **Insight** — רקע / הקשר קריטי.
- **Statement** — ההוראה עצמה.
- **Personality** — סגנון / טון.
- **Experiment** — כמה גרסאות להפיק וכמה הן צריכות להיות שונות זו מזו.

**Template (עברית):**
```
[Capacity + Role]: אתה ___ המתמחה ב-___
[Insight]: הרקע החשוב הוא ___
[Statement]: ההוראה היא ___
[Personality]: הטון צריך להיות ___
[Experiment]: הפק N גרסאות שונות מבחינת ___
```

**דוגמה — יצירת variants לכותרת של סוכן חדש:**

```
Capacity + Role: אתה naming expert המתמחה ב-product naming של כלי AI.
Insight: אני משיקה סוכן שעוזר לבעלי עסקים קטנים לכתוב הצעות מחיר. הקהל שלי הוא פרילנסרים ועצמאים בישראל, בגילאי 30-55, לא טכניים.
Statement: הצע שמות לסוכן.
Personality: שמות חייבים להיות בעברית או טרנסליטרציה אנגלית קלה, קצרים (2-3 מילים), לא צפופים, לא "מפחידים טכנולוגית".
Experiment: הפק 7 שמות שמגוונים ביניהם — 2 משחקי מילים, 2 תיאוריים, 2 מטאפוריים, 1 נועז.
```

**Components Filling Guide:**
- **Capacity + Role** — טוב: תפקיד + תת-התמחות מסוימת ("naming expert המתמחה ב-AI products"). גרוע: "יצירתי".
- **Insight** — טוב: facts על הקהל והקונטקסט שמשפיעים על הפלט (ישראלים 30-55, לא טכניים). גרוע: "לעסקים".
- **Statement** — טוב: הוראה ישירה ("הצע שמות"). גרוע: "מה אתה חושב על...".
- **Personality** — טוב: אילוצי סגנון קונקרטיים (עברית, 2-3 מילים, לא צפוף). גרוע: "סגנון נחמד".
- **Experiment** — טוב: N מדויק + variance מוגדר ("7 שמות: 2 משחקי מילים, 2 תיאוריים, 2 מטאפוריים, 1 נועז"). גרוע: "כמה אופציות".

**Before → After:**
- Before: *"תן לי אופציות לשם למוצר AI שאני משיקה"* — הכל פתוח.
- After: *"Capacity: naming expert ב-AI products. Insight: סוכן לעצמאים 30-55 בישראל, לא טכניים. Statement: הצע שמות. Personality: עברית, 2-3 מילים, לא מפחיד. Experiment: 7 שמות — 2 משחקי מילים, 2 תיאוריים, 2 מטאפוריים, 1 נועז."* — variance מובנה לא מקרי.

## BROKE

**ראשי תיבות:** Background, Role, Objective, Key results, Evolve

**שורה אחת:** פרומפט ל-deliverable עסקי עם KPIs מדידים — מושפע ממתודולוגיית OKRs.

**מתי להשתמש:** המשימה היא business deliverable עם הצלחה מדידה (conversion rate, engagement, retention).

**מתי לא להשתמש:** כשההצלחה סובייקטיבית או איכותית.

**מרכיבים:**
- **Background** — המצב העסקי.
- **Role** — מי אתה (בהקשר עסקי).
- **Objective** — מה המטרה.
- **Key Results** — איך נמדוד הצלחה (מדדים כמותיים).
- **Evolve** — מנגנון ללמידה ושיפור.

**Template (עברית):**
```
[Background]: המצב הוא ___
[Role]: אתה ___
[Objective]: המטרה היא ___
[Key Results]: נמדוד הצלחה לפי: 
  1. ___ 
  2. ___ 
  3. ___
[Evolve]: אחרי כל איטרציה, ___
```

**דוגמה — סוכן שכותב email subject lines:**

```
Background: אני מנהלת newsletter של 8,000 נרשמים בתחום AI. open rate נוכחי: 22%. המתחרים הישירים שלי ב-30%+. אני רוצה להעלות את ה-open rate.
Role: אתה direct response copywriter עם התמחות ב-email subject lines, שעובד data-driven.
Objective: לכתוב 10 גרסאות subject line לכל email שאני שולחת, ממוטבות ל-open rate.
Key Results: 
  1. כל subject line — פחות מ-55 תווים.
  2. אף subject line לא מכיל clickbait או הפרזה שאני לא יכולה לעמוד בה.
  3. מגוון של 4 סגנונות: curiosity, benefit, urgency, personality.
Evolve: אחרי שאני שולחת, אספר לך איזו גרסה ניצחה ומה ה-open rate. שפר את הגישה לגרסאות הבאות על סמך הנתונים.
```

**Components Filling Guide:**
- **Background** — טוב: נתונים כמותיים + gap ביחס לבנצ'מרק (22% נוכחי, 30% מתחרים). גרוע: "רוצה יותר open rate".
- **Role** — טוב: תפקיד + שיטת עבודה (data-driven). גרוע: "קופירייטר טוב".
- **Objective** — טוב: N פעולות + מטרה מדידה ("10 גרסאות, ממוטב ל-open rate"). גרוע: "תעזור לי עם כותרות".
- **Key Results** — טוב: 2-3 מדדים כמותיים + אחד איכותי ספציפי (תווים, ללא clickbait, 4 סגנונות). גרוע: "שיהיה טוב".
- **Evolve** — טוב: מנגנון feedback קונקרטי ("תקבל נתוני winner, עדכן גישה"). גרוע: "נראה איך זה הולך".

**Before → After:**
- Before: *"תכתוב לי כותרות למיילים שיגרמו לאנשים לפתוח"* — אין KPI, אין learning loop.
- After: *"Background: 22% open rate, בנצ'מרק 30%. Role: direct response copywriter data-driven. Objective: 10 גרסאות ממוטבות ל-open rate. KR: <55 תווים, בלי clickbait, 4 סגנונות. Evolve: קבל נתוני winner אחרי כל שליחה."* — מדיד, אופטימיזבילי, משתפר מאיטרציה לאיטרציה.

## TIDD-EC

**ראשי תיבות:** Task, Instructions, Do, Don't, Examples, Context

**שורה אחת:** פרומפט עם כללי compliance מפורשים — רשימות Do ו-Don't נפרדות.

**מתי להשתמש:** יש דרישות מפורשות של "עשה" ו"אל תעשה" שחשוב להפריד לוויזואלית.

**מתי לא להשתמש:** כשהכללים משולבים עם דוגמאות ואין טעם להפרידם — אז CARE.

**מרכיבים:**
- **Task** — מהי המשימה.
- **Instructions** — איך לבצע אותה.
- **Do** — רשימת דברים חיוביים לעשות.
- **Don't** — רשימת דברים לאסור.
- **Examples** — דוגמאות (טוב/רע).
- **Context** — רקע נוסף אם רלוונטי.

**Template (עברית):**
```
[Task]: ___
[Instructions]: ___

[Do]:
- ___
- ___
- ___

[Don't]:
- ___
- ___
- ___

[Examples]:
טוב: ___
רע: ___

[Context]: ___
```

**דוגמה — סוכן שמגיב בקהילת פייסבוק של לקוחות:**

```
Task: כתוב תגובות לשאלות של חברי קהילה בפייסבוק.

Instructions: תגובה צריכה להיות מועילה, קצרה, ומכוונת את השואל לפעולה הבאה.

Do:
- תן תשובה קונקרטית ישירות (לא "תלוי").
- הצע צעד הבא ברור.
- השתמש בלשון "אתה" או "את" לפי מגדר (אם ברור).
- סמן אם יש מסמך/מדריך רלוונטי שכדאי לקרוא.

Don't:
- אל תתנצל ("סליחה שקרה ל-").
- אל תפנה לאדם אחר ("תתכתב עם X").
- אל תוסיף אימוג'ים.
- אל תשתמש בביטויים "מקצוענים" מוגזמים ("בהחלט", "בוודאי").

Examples:
טוב: "הבאג שתיארת הוא ידוע — תיקנו אותו בגרסה 2.4.1 אתמול. עדכני דרך settings → updates ותבדקי אם זה פתור. אם לא, הגיבי כאן ואתמודד."
רע: "סליחה שעברת את זה! בהחלט נבדוק את העניין. תכתבי לי בהודעה פרטית ונתכתב שם."

Context: הקהילה מכילה 2,400 חברים. רובם bootstrappers — מעריכים ישירות, לא "נימוסים".
```

**Components Filling Guide:**
- **Task** — טוב: סוג משימה + הקשר ערוץ ("כתוב תגובות לשאלות בקהילת פייסבוק"). גרוע: "תכתוב תגובות".
- **Instructions** — טוב: עקרון מנחה על-כללי אחד ("מועילה, קצרה, מכוונת לפעולה"). גרוע: instructions ארוכים שחוזרים על ה-Do/Don't.
- **Do** — טוב: פעולות ספציפיות הניתנות למדידה ("תן תשובה קונקרטית, לא 'תלוי'", "הצע צעד הבא ברור"). גרוע: "תהיה מועיל" (איך מודדים?).
- **Don't** — טוב: התנהגויות שמשתמש מזהה מיידית ("אל תתנצל ב'סליחה שקרה'", "אל תוסיף אימוג'ים"). גרוע: "אל תהיה לא מקצועי" (כללי מדי).
- **Examples** — טוב: זוג טוב/רע של אותה סיטואציה, כדי שההבדל ברור. גרוע: דוגמה טובה לבד (קלוד לא יודע מהו הגבול).
- **Context** — טוב: facts על הקהל/הערוץ שמשנים את הטון ("bootstrappers, מעריכים ישירות"). גרוע: "קהילה של אנשים".

**Before → After:**
- Before: *"תכתוב תגובות טובות בפייסבוק, תהיה מקצועי אבל נחמד"* — אין הבחנה בין כן/לא, אין דוגמה, אין הקשר.
- After: *"Task: תגובות בקהילת פייסבוק. Do: קונקרטי, צעד הבא, לשון אישית. Don't: התנצלות מוגזמת, אימוג'ים, הפניה לאדם אחר. Example טוב/רע: [תגובה ישירה עם פתרון] vs [תגובה מתנצלת]. Context: 2,400 bootstrappers, מעריכים ישירות."* — אכיפה ברורה, המודל יודע איפה קו הגבול.

## CARE

**ראשי תיבות:** Context, Ask, Rules, Examples

**שורה אחת:** פרומפט עם כללים ודוגמאות משולבים — Rules + Examples כיחידה.

**מתי להשתמש:** הכללים והדוגמאות קשורים זה לזה, ואין טעם להפרידם.

**מתי לא להשתמש:** כשחשוב להפריד Do מ-Don't — אז TIDD-EC.

**מרכיבים:**
- **Context** — המצב / הרקע.
- **Ask** — הבקשה הספציפית.
- **Rules** — כללים.
- **Examples** — דוגמאות שממחישות את הכללים.

**Template (עברית):**
```
[Context]: ___
[Ask]: ___

[Rules]:
1. ___
2. ___
3. ___

[Examples]:
- ___
- ___
- ___
```

**דוגמה — סוכן שכותב descriptions לסקילים:**

```
Context: אני בונה suite של skills פרטיים ב-Claude. descriptions של skills חייבים להיות כתובים בצורה שתפעיל את הסקיל במצבים הנכונים בלבד — לא יותר מדי רגיש ולא פחות מדי.

Ask: כתוב description לסקיל חדש שאני מתאר.

Rules:
1. כל description מתחיל בפעולה/תוצאה ראשית במשפט אחד.
2. כולל 5-8 ביטויי טריגר מגוונים (עברית ואנגלית).
3. כולל גם trigger מפורש (משתמש מבקש) וגם trigger סמוי (משתמש מתאר מצב שהסקיל מתאים לו).
4. ללא ז'רגון טכני. המשתמש הסופי הוא לא מפתח.
5. סוף עם "הפעל את הסקיל" או "השתמש בסקיל הזה" — לא "use this skill".

Examples:
- description טוב של humanizer-he מתחיל ב: "זיהוי דפוסי כתיבה AI בטקסט עברי, ושכתובם לעברית טבעית."
- description לא טוב: "This skill helps with Hebrew text processing and provides various improvements to AI-generated content."
```

**Components Filling Guide:**
- **Context** — טוב: הקשר שמסביר *למה* הכללים חשובים ("descriptions של skills צריכים להפעיל את הסקיל במצבים הנכונים בלבד"). גרוע: "אני צריכה כלל".
- **Ask** — טוב: פעולה אחת ברורה ("כתוב description"). גרוע: רשימת כמה משימות ("תכתוב description ותתן לי דוגמאות ותסכם").
- **Rules** — טוב: 3-5 כללים ספציפיים ומדידים, ממוספרים. גרוע: pep talk ("תהיה יצירתי", "אל תהיה משעמם").
- **Examples** — טוב: 2-3 דוגמאות שכל אחת ממחישה כלל אחר מהרשימה. גרוע: דוגמה אחת שלא מראה את כל הכללים בפעולה.

**Before → After:**
- Before: *"תכתוב לי description טוב לסקיל. שיהיה ברור ולא שיווקי"* — כללים עמומים, אין דוגמאות, אין הקשר.
- After: *"Context: skill description מפעיל את הסקיל. Ask: כתוב description לסקיל [X]. Rules: 1) משפט ראשון תמציתי, 2) 5-8 טריגרים, 3) עברית+אנגלית, 4) בלי ז'רגון, 5) סיום ב'הפעל'. Examples: [טוב] vs [גרוע]."* — הכללים אכיפים, הדוגמאות ממחישות.

## CO-STAR

**ראשי תיבות:** Context, Objective, Style, Tone, Audience, Response

**שורה אחת:** הפריימוורק המובנה ביותר למשימות כתיבה ותקשורת — קהל, טון וסגנון במרכז.

**מתי להשתמש:** משימת כתיבה/תקשורת שבה קהל, טון, וסגנון קריטיים. זה הפריימוורק לבחור אם הפלט הוא content.

**מתי לא להשתמש:** משימות טכניות (קוד, data) שבהן הטון פחות חשוב — אז RISEN.

**מרכיבים:**
- **Context** — הרקע/המצב.
- **Objective** — המטרה של הפלט.
- **Style** — סגנון כתיבה.
- **Tone** — טון רגשי.
- **Audience** — הקהל הקונקרטי.
- **Response** — פורמט התגובה.

**Template (עברית):**
```
# Context
___

# Objective
___

# Style
___

# Tone
___

# Audience
___

# Response
___
```

**דוגמה — סוכן שכותב LinkedIn posts לייסדת startup:**

```
# Context
אני ייסדת משותפת של סטארטאפ AI בן 8 חודשים. 180 משתמשים בחינם, 12 משלמים. גייסנו pre-seed בשווי $2M. אני מחפשת לבנות נוכחות ב-LinkedIn לקראת raise של seed.

# Objective
לכתוב LinkedIn post שיגרום לאנשים ב-AI community לעקוב אחריי, ויתחיל שיחות שיובילו למכירות או לקשרי VC.

# Style
סיפורי (narrative) — מתחיל מנקודת כאב, עובר לתובנה, מסתיים בשאלה פתוחה. קצר מספיק להיראות בלי "see more" (מתחת ל-200 מילים).

# Tone
חם, כן, לא מתנצל ולא מתגאה. כמו ייסדת מוכשרת שמדברת עם עמיתים במשקה אחרי העבודה.

# Audience
מייסדי SaaS ו-AI בשנים הראשונות, VCs של seed/series A, אנשי דאטה וממ"ע בחברות בינוניות שמתעניינים ב-AI.

# Response
Post בפורמט LinkedIn (שורות קצרות, רווח בין פסקאות, לא יותר מ-8 שורות של טקסט רצוף). ללא אימוג'ים, ללא hashtags.
```

**Components Filling Guide:**
- **Context** — טוב: facts על המצב + היסטוריה + איפה הפלט יופיע ("ייסדת 8 חודשים, 180 משתמשים, pre-seed"). גרוע: "יש לי סטארטאפ".
- **Objective** — טוב: פעולה + מה היא אמורה להשיג ("Post שיגרום לאנשים לעקוב + יתחיל שיחות שיובילו למכירות או VCs"). גרוע: "פוסט טוב".
- **Style** — טוב: תיאור מבני (narrative 3-act, אורך מדויק, פורמט). גרוע: "סגנון מקצועי".
- **Tone** — טוב: emotion + comparison ("כמו ייסדת מוכשרת במשקה אחרי העבודה"). גרוע: "ידידותי".
- **Audience** — טוב: 2-3 sub-segments עם role+size ("מייסדי SaaS, VCs של seed, אנשי דאטה"). גרוע: "אנשי AI".
- **Response** — טוב: פורמט ספציפי + constraints נגטיביים ("שורות קצרות, <8 שורות רצופות, בלי אימוג'ים"). גרוע: "פוסט יפה".

**Before → After:**
- Before: *"תכתוב לי פוסט LinkedIn על זה שאני בונה סטארטאפ ושאני רוצה שאנשים יעקבו אחריי"* — 6 רכיבים חסרים.
- After: *"Context: ייסדת startup 8 חודשים, 180 users, לקראת seed. Objective: עקיבה + שיחות עם VCs. Style: narrative 3-act, <200 מילים. Tone: כמו שיחה אחרי העבודה. Audience: מייסדי SaaS, VCs, דאטה. Response: LinkedIn format, שורות קצרות, ללא אימוג'ים."* — כל רכיב מגובש, אין פער ליישום.

## RISEN

**ראשי תיבות:** Role, Instructions, Steps, End-goal, Narrowing

**שורה אחת:** הפריימוורק המובנה ביותר לפרוצדורה רב-שלבית — Steps + End-goal + Narrowing מבטיחים ביצוע מדויק.

**מתי להשתמש:** משימה היא מתודולוגיה רב-שלבית. זה פריימוורק ליבה לבניית סוכנים — רוב ה-system prompts של סוכנים הם RISEN.

**מתי לא להשתמש:** משימת כתיבה פשוטה ללא steps — אז CO-STAR או RTF.

**מרכיבים:**
- **Role** — מי אתה.
- **Instructions** — הוראות כלליות לביצוע.
- **Steps** — רצף מדויק של צעדים (ממוספר).
- **End-goal** — התוצאה הסופית הרצויה.
- **Narrowing** — מגבלות ואילוצים (לצמצם את מרחב הפלטים האפשריים).

**Template (עברית):**
```
# Role
אתה ___

# Instructions
___

# Steps
1. ___
2. ___
3. ___
...

# End-goal
בסיום, צריך להיות: ___

# Narrowing
אילוצים:
- ___
- ___
```

**דוגמה — סוכן שעושה market research על מתחרה:**

```
# Role
אתה competitive intelligence analyst עם 10 שנות ניסיון ב-B2B SaaS, המתמחה בניתוח strategic positioning.

# Instructions
המשתמש יתן לך שם של מתחרה. אתה תבצע ניתוח מובנה של הפוזיציה שלו בשוק, בהתבסס על מקורות פומביים.

# Steps
1. חפש את האתר הראשי של המתחרה וקרא את homepage ו-pricing page.
2. חפש את ה-LinkedIn של המתחרה, ודוק תיאור, מספר עובדים, trend של גיוסים.
3. חפש 3 ביקורות של לקוחות (G2 / Capterra / Reddit) — לפחות אחת חיובית ואחת שלילית.
4. זהה: target ICP, core value prop, מודל תמחור, ו-3 חולשות שעולות מהביקורות.
5. השווה לחברה שלנו (שתתואר בהקשר) והצג differentiators.

# End-goal
דוח של 400-600 מילים הכולל: ICP, value prop, pricing, 3 חולשות, 2-3 הזדמנויות דיפרנציאציה לנו.

# Narrowing
- אל תשתמש במקורות מעבר ל-3 שנים.
- אל תמציא נתוני כספים אם לא מצאת — ציין "לא נמצא במקורות פומביים".
- אל תתן המלצות פעולה — רק ניתוח.
- עברית, אבל שמות מוצרים ו-titles מקוריים.
```

**Components Filling Guide:**
- **Role** — טוב: תפקיד + שנות ניסיון + תת-תחום ("competitive intelligence analyst, 10 שנים ב-B2B SaaS"). גרוע: "אנליסט".
- **Instructions** — טוב: משפט או שניים שמגדירים *סוג* הפלט בתור עקרון ("ניתוח מובנה מבוסס מקורות פומביים"). גרוע: rewrite של ה-Steps.
- **Steps** — טוב: 5-7 צעדים ספציפיים עם פעולה + מקור ("1. חפש website, קרא homepage+pricing. 2. חפש LinkedIn, תעד size+trend. 3. ..."). גרוע: "תחקור את המתחרה".
- **End-goal** — טוב: deliverable מדיד + תחומים חובה ("דוח 400-600 מילים: ICP, value prop, pricing, 3 חולשות, 2-3 הזדמנויות"). גרוע: "תן לי מידע טוב".
- **Narrowing** — טוב: 3-5 איסורים ספציפיים שמונעים failure modes (אל תמציא, אל תמליץ, אל תשתמש במקורות ישנים). גרוע: "שיהיה איכותי".

**Before → After:**
- Before: *"תעזור לי לעשות ניתוח מתחרים של [חברה]. תבדוק הכל ותן לי תמונה מלאה"* — אין מבנה, אין deliverable ברור, אין guard rails.
- After: *"Role: CI analyst 10 שנים ב-B2B SaaS. Instructions: ניתוח מבוסס מקורות פומביים. Steps: 5 צעדים ספציפיים. End-goal: דוח 400-600 מילים עם 5 חלקים מוגדרים. Narrowing: לא להמציא, לא להמליץ, לא להסתמך על מעל 3 שנים."* — סוכן יודע איפה להתחיל, מה לייצר, ומה אסור.

**Research basis:** RISEN מוכר בקהילת practitioner-prompting (Vikram Sharma ואחרים, 2023-2024) כפריימוורק ליבה ל-procedural prompting. אין מאמר אקדמי מוגדר, אבל המרכיבים (Role, sequential Steps, End-goal, negative constraints) נתמכים באופן רוחבי ב-literature של instruction-following (Wei et al., *Chain-of-Thought*, NeurIPS 2022; Madaan et al., *Self-Refine*, NeurIPS 2023).

## RISE-IE

**ראשי תיבות:** Role, Input, Steps, Expectations (variant: Input-Expectation)

**שורה אחת:** RISE בגרסת טרנספורמציית נתונים — input ברור → steps → output מצופה.

**מתי להשתמש:** משימה של המרת נתונים (CSV ל-JSON, טבלה לסיכום, קוד לתיעוד). יש input מובנה ויש expectation מוגדרת.

**מתי לא להשתמש:** משימה יצירתית ללא input מובנה — אז RISE-IX.

**מרכיבים:**
- **Role** — מי אתה.
- **Input** — תיאור מדויק של הקלט (פורמט, מבנה, characteristics).
- **Steps** — שלבי הטרנספורמציה.
- **Expectations** — תיאור מדויק של הפלט המצופה.

**Template (עברית):**
```
# Role
___

# Input
הקלט יהיה ___
פורמט: ___
מאפיינים: ___

# Steps
1. ___
2. ___

# Expectations
הפלט צריך:
- להיות בפורמט ___
- לכלול ___
- לא לכלול ___
```

**דוגמה — סוכן ממיר תמלולי שיחות ל-structured data:**

```
# Role
אתה data engineer המתמחה בחילוץ מידע מובנה מדיאלוגים.

# Input
הקלט יהיה תמלול של שיחת מכירה בעברית.
פורמט: טקסט חופשי, ללא סימני דוברים רשמיים. לעיתים יש סימנים כמו "שואל:" או "לקוח:".
מאפיינים: 20-60 דקות, 2-4 משתתפים, שפה דיבורית עם שיבושים ומילוי.

# Steps
1. זהה את המשתתפים וסיווג (sales rep / prospect / silent observer).
2. חלץ את ה-pain points שהלקוח הפוטנציאלי הזכיר.
3. חלץ את ה-objections שהוא העלה.
4. חלץ את ה-next steps שהוסכם.
5. חלץ את רמת המעורבות (engagement level) של הלקוח: low / medium / high.

# Expectations
הפלט צריך:
- להיות בפורמט JSON תקין.
- לכלול: participants (array), pain_points (array), objections (array), next_steps (array), engagement_level (string).
- לא לכלול: תוכן המילולי המלא של השיחה, תקציר נרטיבי, המלצות.
```

**Components Filling Guide:**
- **Role** — טוב: תפקיד טכני + התמחות ("data engineer המתמחה בחילוץ מדיאלוגים"). גרוע: "analyst".
- **Input** — טוב: פורמט + characteristics + edge cases ("טקסט חופשי, לפעמים עם 'שואל:' או 'לקוח:', 20-60 דקות, שיבושים"). גרוע: "יש לי תמלול".
- **Steps** — טוב: רצף של חילוצים עם קריטריון ברור לכל אחד ("זהה משתתפים וסווג, חלץ pain points, חלץ objections..."). גרוע: "תחלץ את המידע".
- **Expectations** — טוב: schema מלא + 'לכלול' + 'לא לכלול' ("JSON עם 5 שדות, בלי תקציר נרטיבי, בלי המלצות"). גרוע: "תחזיר JSON מסודר".

**Before → After:**
- Before: *"יש לי תמלול שיחת מכירה, תחלץ לי מידע שימושי כ-JSON"* — input לא מוגדר, steps לא מוגדר, expected schema לא מוגדר.
- After: *"Role: data engineer. Input: תמלול חופשי, 20-60 דקות, לפעמים סימני דוברים. Steps: 5 חילוצים ספציפיים. Expectations: JSON עם 5 שדות מוגדרים, בלי נרטיב, בלי המלצות."* — output דטרמיניסטי, ניתן לאינטגרציה עם pipeline.

## RISE-IX

**ראשי תיבות:** Role, Instructions, Steps, eXamples (variant: Instructions-Examples)

**שורה אחת:** RISE בגרסת יצירה יצירתית — few-shot learning עם דוגמאות רפרנס.

**מתי להשתמש:** משימה יצירתית שבה יש דוגמאות של פלט טוב, והן מלמדות את הדפוס טוב יותר מהוראות מילוליות.

**מתי לא להשתמש:** אין לך דוגמאות, או שהן מאוד שונות מה-output הרצוי — אז RISEN או CO-STAR.

**מרכיבים:**
- **Role** — מי אתה.
- **Instructions** — הוראות כלליות.
- **Steps** — תהליך לביצוע.
- **Examples** — 2-5 דוגמאות של input→output.

**Template (עברית):**
```
# Role
___

# Instructions
___

# Steps
1. ___
2. ___

# Examples

## דוגמה 1
Input: ___
Output: ___

## דוגמה 2
Input: ___
Output: ___
```

**דוגמה — סוכן שכותב taglines למוצרי AI:**

```
# Role
אתה copywriter המתמחה ב-B2B SaaS taglines.

# Instructions
תן tagline אחד של 4-7 מילים בעברית, שמתאר את הערך הראשי של המוצר ומשדר אישיות.

# Steps
1. זהה את הערך הקונקרטי ביותר שהמוצר מספק.
2. זהה מילה אחת שעושה את התיאור ייחודי (לא clichéית).
3. כתוב tagline שמשלב את שניהם.

# Examples

## דוגמה 1
Input: "פלטפורמה שבה צוותי שיווק יכולים לבנות הודעות דיוור מותאמות אישית בעזרת AI, בלי לתכנת."
Output: "דיוור אישי. בלי קוד. בלי חשבון."

## דוגמה 2
Input: "כלי שמזהה אנומליות בלוגים של שרתים בזמן אמת ושולח התראות."
Output: "הלוגים מדברים. אנחנו מקשיבים."

## דוגמה 3
Input: "אפליקציה שעוזרת למורים להכין שיעורים מהיר יותר."
Output: "השיעור הבא שלך — מוכן עד ההפסקה."
```

**Components Filling Guide:**
- **Role** — טוב: תפקיד מוגדר צר ("copywriter המתמחה ב-B2B SaaS taglines"). גרוע: "כותב".
- **Instructions** — טוב: constraints ברורים (אורך, שפה, רוח). גרוע: "תן tagline טוב".
- **Steps** — טוב: רצף קצר ופעולתי (2-3 steps). ב-RISE-IX ה-Examples עושים רוב העבודה, לא ה-Steps. גרוע: steps ארוכים שכופלים את ה-Examples.
- **Examples** — טוב: 2-4 examples שמכסים את מרחב הסגנונות הרצוי, כל אחד input+output קונקרטי. גרוע: דוגמה אחת (לא מספיק ל-pattern recognition), או 8+ (רעש מיותר).

**Before → After:**
- Before: *"תכתוב tagline למוצר: [תיאור]. תסתכל על דוגמאות של taglines טובים באינטרנט"* — Claude לא יודע *איזה* סגנון אתה רוצה.
- After: *"Role: copywriter B2B SaaS. Instructions: tagline 4-7 מילים עברית, אישיות + ערך ראשי. Examples: [3 דוגמאות input→output]"* — Claude מסיק את הדפוס מהדוגמאות, מדויק יותר.

**Research basis:** Few-shot prompting — **Brown et al., *Language Models are Few-Shot Learners*, NeurIPS 2020** (הידוע כ-GPT-3 paper). הוכח שהוספת 2-5 דוגמאות טובות משפרת ביצועים מעל הוראות מילוליות ארוכות.

## BAB

**ראשי תיבות:** Before, After, Bridge

**שורה אחת:** פרומפט לשכתוב/המרה חד-פעמית — "זה המצב הנוכחי, זה המצב הרצוי, הנה כללי הגשר".

**מתי להשתמש:** יש תוכן קיים שצריך להמיר למצב חדש, וכללי ההמרה ברורים.

**מתי לא להשתמש:** שיפור איטרטיבי (אז Self-Refine), או יצירה מאפס (אז CREATE category).

**מרכיבים:**
- **Before** — התוכן הנוכחי + מה הבעיות בו.
- **After** — התוכן הרצוי + מה המאפיינים שלו.
- **Bridge** — כללי הטרנספורמציה.

**Template (עברית):**
```
# Before (מצב נוכחי)
[תוכן נוכחי]
הבעיות: ___

# After (מצב רצוי)
תוכן שיש לו את המאפיינים הבאים: ___

# Bridge (כללי טרנספורמציה)
1. ___
2. ___
3. ___
```

**דוגמה — שיפור system prompt קיים של סוכן:**

```
# Before (מצב נוכחי)
[הפרומפט הקיים מודבק כאן]
הבעיות: הסוכן נותן תשובות ארוכות מדי, מתחיל כל תשובה ב"כמובן!", ולא שואל שאלות הבהרה גם כשצריך.

# After (מצב רצוי)
פרומפט שגורם לסוכן: לקצר תשובות (1-3 משפטים למשימות פשוטות), להפסיק עם "כמובן" ודומותיה, ולשאול שאלה אחת ממוקדת אם המידע חסר.

# Bridge (כללי טרנספורמציה)
1. הוסף לחלק "Narrowing" כלל מפורש על אורך תשובות.
2. הוסף רשימת ביטויים אסורים בפתיחה.
3. הוסף הוראה: "לפני ביצוע, שאל שאלה אחת אם חסר לך X, Y, או Z."
4. שמור את שאר המבנה של הפרומפט — רק עדן.
5. שמור את השפה (עברית/אנגלית) כפי שהיא.
```

**Components Filling Guide:**
- **Before** — טוב: תוכן מקורי + רשימת בעיות ספציפיות ("ארוך מדי, מתחיל ב'כמובן', לא שואל"). גרוע: "זה לא טוב".
- **After** — טוב: מאפיינים של התוצאה הרצויה, לא תוצאה ספציפית ("פרומפט שגורם לסוכן לקצר, להפסיק ביטויים אסורים, לשאול כשחסר"). גרוע: דוגמה של תוצאה סופית — זה משאיר פחות מקום להתאמה.
- **Bridge** — טוב: 3-5 כללי טרנספורמציה ממוספרים, כל אחד פעיל וספציפי ("הוסף כלל על אורך, הוסף רשימת ביטויים אסורים, הוסף הוראת clarify"). גרוע: "תשפר".

**Before → After:**
- Before: *"יש לי system prompt שלא עובד טוב. תעזור לי לשפר אותו"* — אין הגדרת היעד, אין כללי המרה.
- After: *"Before: [פרומפט], בעיות: ארוך מדי, פותח ב'כמובן', לא שואל. After: פרומפט שגורם לסוכן לקצר/להפסיק ביטויים/לשאול. Bridge: 1) הוסף כלל אורך, 2) הוסף ביטויים אסורים, 3) הוסף clarify instruction, 4) שמור שפה."* — טרנספורמציה דטרמיניסטית.

## Self-Refine

**ראשי תיבות:** (אין ראשי תיבות — שם שלם)

**שורה אחת:** לולאת שיפור איטרטיבית — Generate → Feedback → Refine, חוזר עד עמידה בקריטריון.

**מתי להשתמש:** האיכות חשובה, אין דחיפות, וערך גבוה של שיפור הדרגתי.

**מתי לא להשתמש:** משימה חד-פעמית או שהשיפור האיטרטיבי overhead.

**מרכיבים:**
- **Initial generation** — הפלט הראשוני.
- **Feedback dimensions** — על מה להעריך.
- **Refinement loop** — איך לשפר.
- **Stop condition** — מתי להפסיק.

**Template (עברית):**
```
בצע את המשימה הבאה בלולאת שיפור עצמית:

# Task
___

# Initial Generation
יצר פלט ראשוני.

# Feedback Dimensions
לאחר מכן, בקר את הפלט שלך לפי:
1. ___
2. ___
3. ___

# Refinement
שפר את הפלט על סמך הביקורת. חזור על Feedback + Refinement עד להגעה לקריטריון העצירה.

# Stop Condition
עצור כאשר: ___
או אחרי N איטרציות — אחת מהשתיים, המוקדמת יותר.

# Output
הצג את כל האיטרציות, וסמן את הסופית.
```

**דוגמה — שיפור איטרטיבי של description לסקיל:**

```
# Task
כתוב description לסקיל "meeting-summarizer" שמסכם פגישות.

# Initial Generation
יצר description ראשוני לפי הכללים של סקילים (ראשי + טריגרים + closing).

# Feedback Dimensions
1. האם יש לפחות 6 ביטויי טריגר?
2. האם הביטויים מגוונים — גם עברית גם אנגלית, גם בקשה ישירה גם תיאור מצב?
3. האם המשפט הראשון חד ומתאר בפועל מה הסקיל עושה?
4. האם הסוף אומר "הפעל את הסקיל" / "השתמש בסקיל הזה"?

# Refinement
שפר על סמך הפערים. הרץ Feedback + Refinement שוב.

# Stop Condition
עצור כשכל 4 הממדים עונים "כן" באיטרציה אחת, או אחרי 4 איטרציות.

# Output
הצג את description של כל איטרציה, והסבר מה השתנה.
```

**Components Filling Guide:**
- **Initial Generation** — טוב: הוראה פשוטה בלי אילוצי feedback כבר בשלב הזה ("יצר פלט ראשוני לפי הכללים"). גרוע: להטעין את ה-generation עם כל הקריטריונים (זה תפקיד ה-Feedback).
- **Feedback Dimensions** — טוב: 3-5 ממדים בינאריים או מדורגים שניתן לבדוק ("האם יש 6 טריגרים? האם יש גם עברית וגם אנגלית?"). גרוע: ממדים עמומים ("האם זה טוב?", "האם זה מקצועי?").
- **Stop Condition** — טוב: קריטריון נצפה + upper bound של איטרציות ("כל הממדים 'כן', או אחרי 4"). גרוע: "עד שזה טוב".

**Before → After:**
- Before: *"תכתוב description טוב. אם זה לא מספיק, תשפר."* — אין feedback dimensions, אין stop condition.
- After: *"Initial: יצר פלט ראשוני. Feedback: 4 ממדים בינאריים [רשימה]. Refinement: שפר על סמך הפערים. Stop: כל 'כן' או אחרי 4 איטרציות."* — loop מובנה, מתכנס, מדיד.

**Research basis:** **Madaan et al., *Self-Refine: Iterative Refinement with Self-Feedback*, NeurIPS 2023** (arXiv 2303.17651). הוכח שיפור ממוצע של ~20% על task benchmarks כאשר המודל מבקר ומשפר את הפלט של עצמו איטרטיבית.

## Chain of Density

**ראשי תיבות:** (אין — שם שלם)

**שורה אחת:** דחיסה איטרטיבית — מייצר סיכום, ואז מעבה אותו עם מידע נוסף בלי להאריך.

**מתי להשתמש:** סיכום תוכן ארוך שחייב להיות גם קצר וגם עשיר במידע.

**מתי לא להשתמש:** סיכום רגיל — זה overkill.

**מרכיבים:**
- **Content** — התוכן לסכם.
- **Length target** — כמה מילים/משפטים.
- **Density iterations** — N איטרציות של הצפפה.
- **Stopping criterion** — מתי להפסיק.

**Template (עברית):**
```
סכם את התוכן הבא בטכניקת Chain of Density:

# Content
___

# Length
N מילים בדיוק (אורך קבוע לכל איטרציה).

# Process
1. איטרציה 1: סיכום אוורירי (~4-5 entities/concepts).
2. איטרציה 2-5: בכל איטרציה, זהה 1-3 concepts חדשים חשובים שלא הוזכרו, והוסף אותם — בלי להאריך מעבר ל-N.
3. להשיג זאת: הסר מילים מיותרות, שלב משפטים, החלף ביטויים כלליים ב-specifics.

# Output
הצג את 5 הגרסאות. האחרונה היא הצפופה ביותר.
```

**דוגמה — סיכום תיעוד של סוכן קיים:**

```
# Content
[תיעוד 3,000 מילים של סוכן שבנית]

# Length
80 מילים בכל איטרציה.

# Process
איטרציה 1: 80 מילים, 4 concepts (מי הסוכן, למה נועד, איך עובד, מה ההצלחה).
איטרציה 2: 80 מילים, 6 concepts (הוסף 2 specifics).
איטרציה 3: 80 מילים, 8 concepts.
איטרציה 4: 80 מילים, 10 concepts.
איטרציה 5: 80 מילים, 12 concepts — הגרסה הכי צפופה.

# Output
הצג את 5 האיטרציות. בחר את האיטרציה האחרונה לקובץ README.md של הסוכן.
```

**Components Filling Guide:**
- **Content** — טוב: תוכן מוגדר היקף ברור ("תיעוד 3,000 מילים של סוכן"). גרוע: "קצת מידע".
- **Length target** — טוב: N מילים מדויק, זהה בכל האיטרציות ("80 מילים"). גרוע: "קצר" או "בערך 100" — פותח דלת לרגרסיה.
- **Density iterations** — טוב: מספר מוגדר (5 הוא הסטנדרט מהמחקר). גרוע: "עד שיתייצב" (לא דטרמיניסטי).

**Before → After:**
- Before: *"תסכם לי את התיעוד הזה בצורה צפופה ומועילה"* — אין אורך, אין מספר איטרציות, אין קריטריון לצפיפות.
- After: *"Content: [תיעוד]. Length: 80 מילים קבוע. Process: 5 איטרציות של הצפפה. Each iteration: שמור אורך קבוע, הוסף 1-3 concepts חדשים."* — תהליך מובנה שמתכנס לסיכום דחוס פי כמה.

**Research basis:** **Adams et al., *From Sparse to Dense: GPT-4 Summarization with Chain of Density Prompting*, Columbia + Salesforce AI, 2023** (arXiv 2309.04269). הטכניקה המקורית — 5 איטרציות של הוספת entities חדשות תוך שמירה על אורך קבוע.

## Skeleton of Thought

**ראשי תיבות:** (אין — שם שלם)

**שורה אחת:** שלד ראשון, הרחבה אחר כך — מייצר outline, ואז ממלא כל נקודה.

**מתי להשתמש:** תוכן ארוך שיכול להיכתב במקביל אחרי שיש לו outline.

**מתי לא להשתמש:** תוכן קצר, או תוכן שדורש זרימה רציפה שלא מתאימה לחלוקה.

**מרכיבים:**
- **Topic** — נושא.
- **Skeleton points** — כמה נקודות במבנה.
- **Expansion depth** — כמה להרחיב כל נקודה.

**Template (עברית):**
```
# Task
כתוב [סוג התוכן] על הנושא: ___

# Phase 1 — Skeleton
ראשית, צור שלד של N נקודות. כל נקודה — שורה אחת בלבד (כותרת + משפט תיאור).

# Phase 2 — Expansion
לאחר מכן, לכל נקודה בנפרד, כתוב [X מילים / Y משפטים / Z פסקאות].

# Phase 3 — Assembly
חבר את הכל ל-output אחד. אל תערוך חיבורים בין הנקודות — הקורא אמור לראות כל נקודה כיחידה עצמאית.
```

**דוגמה — כתיבת מדריך על בניית סקיל:**

```
# Task
כתוב מדריך על "איך לבנות skill פרטי ב-Claude שעובד מצוין".

# Phase 1 — Skeleton
צור שלד של 7 נקודות. כל נקודה — כותרת + משפט תיאור.

# Phase 2 — Expansion
לכל אחת מה-7 נקודות, כתוב 80-120 מילים: מה הנקודה, דוגמה, tip מעשי.

# Phase 3 — Assembly
חבר הכל. השתמש ב-headers, השאר הפרדה ויזואלית בין נקודות.
```

**Components Filling Guide:**
- **Topic** — טוב: נושא + היקף מוגדר ("איך לבנות skill פרטי ב-Claude שעובד מצוין"). גרוע: "תכתוב על skills".
- **Skeleton points** — טוב: N מדויק + הנחיה לכותרת+משפט תיאור ("7 נקודות, כל אחת כותרת+משפט"). גרוע: "כמה נקודות טובות".
- **Expansion depth** — טוב: אורך + מבנה פנימי מוגדר ("80-120 מילים: נקודה + דוגמה + tip מעשי"). גרוע: "תרחיב".

**Before → After:**
- Before: *"תכתוב מדריך מפורט על בניית skills"* — אין מבנה, אין אורך, אין נק' מוגדרות.
- After: *"Phase 1: שלד של 7 נקודות, כותרת+משפט. Phase 2: הרחב כל אחת ל-80-120 מילים עם נקודה+דוגמה+tip. Phase 3: חיבור עם headers."* — outline-first, כל חלק דטרמיניסטי.

**Research basis:** **Ning et al., *Skeleton-of-Thought: Large Language Models Can Do Parallel Decoding*, ICLR 2024** (arXiv 2307.15337). הטכניקה הותאמה לתהליך סדרתי (outline→expand) והוכחה כמקצרת latency וגם משפרת coherence ב-long-form generation.

## Plan-and-Solve (PS+)

**ראשי תיבות:** Plan + Solve (plus enhancements)

**שורה אחת:** Zero-shot לחישובים — "תכנן קודם, פתור אחר כך, ציין משתנים".

**מתי להשתמש:** בעיית חישוב מספרי, zero-shot (אין דוגמאות). יותר מדויק מ-CoT רגיל למספרים.

**מתי לא להשתמש:** בעיה לא מספרית.

**מרכיבים:**
- **Trigger phrase** — "בוא נתכנן את הפתרון ונפתור שלב-שלב. שים לב לשמות משתנים ולערכים המספריים."
- **Variable extraction** — לזהות ולרשום כל המשתנים.
- **Step-by-step solve** — לבצע כל שלב.

**Template (עברית):**
```
# Problem
___

# Instructions
בוא נתכנן את הפתרון ונפתור שלב-שלב. שים לב לשמות משתנים ולערכים המספריים.

1. זהה את כל המשתנים בבעיה והרשום את ערכיהם.
2. תכנן את רצף החישובים.
3. בצע כל חישוב בנפרד, תוך ציון המשתנה ותוצאה.
4. ציין את התשובה הסופית בבירור.
```

**דוגמה — חישוב עלות rate limits לסוכן:**

```
# Problem
אני בונה סוכן ללקוחות. כל לקוח שולח בממוצע 80 הודעות ביום. כל הודעה צורכת בממוצע 4,000 tokens input + 1,200 tokens output. יש לי 35 לקוחות. Claude Sonnet 4.5 עולה $3/1M input ו-$15/1M output. חשב את העלות החודשית.

# Instructions
בוא נתכנן את הפתרון ונפתור שלב-שלב. שים לב לשמות משתנים ולערכים המספריים.

1. זהה את כל המשתנים: messages_per_user_per_day, input_tokens_per_message, output_tokens_per_message, num_users, input_rate, output_rate.
2. תכנן: (a) חשב הודעות יומיות כוללות, (b) חשב input tokens יומי וחודשי, (c) חשב output tokens יומי וחודשי, (d) חשב עלות input ו-output, (e) סכום.
3. בצע כל חישוב בנפרד.
4. ציין את התשובה הסופית ב-$/חודש.
```

**Components Filling Guide:**
- **Problem** — טוב: תיאור מלא + כל הנתונים המספריים + תלויות ידועות ("80 הודעות/יום, 4,000 input tokens, 1,200 output, 35 לקוחות, $3 input, $15 output"). גרוע: "כמה זה יעלה לי?".
- **Trigger phrase** — השתמש בניסוח מדויק: "בוא נתכנן את הפתרון ונפתור שלב-שלב. שים לב לשמות משתנים ולערכים המספריים." זה הטריגר שמעלה דיוק ב-zero-shot.
- **Instructions** — טוב: 4 שלבים: identify variables → plan sequence → solve each → state final answer. גרוע: "תחשב".

**Before → After:**
- Before: *"כמה יעלה לי להריץ סוכן ללקוחות אם יש לי 35 לקוחות?"* — חסר מידע, אין הוראת planning.
- After: *"Problem: 80 הודעות/יום, 4K input + 1.2K output per message, 35 users, $3/$15 pricing. Instructions: [trigger phrase + 4 שלבים]."* — החישוב שקוף, ניתן לאימות, פחות hallucinations של מספרים.

**Research basis:** **Wang et al., *Plan-and-Solve Prompting*, ACL 2023** (arXiv 2305.04091). הוכח שיפור של עד 11% ב-math benchmarks (GSM8K) מעל Chain-of-Thought רגיל, על ידי הוספת שלב תכנון מפורש לפני הפתרון.

## Least-to-Most

**ראשי תיבות:** (אין — שם שלם)

**שורה אחת:** פירוק לתתי-בעיות מסודרות בתלויות — פותר מהקטנה לגדולה.

**מתי להשתמש:** בעיה מורכבת שבה יש תלויות ברורות (B תלוי ב-A, C תלוי ב-B).

**מתי לא להשתמש:** אין תלויות ברורות, או שהבעיה לא מתפרקת טבעית.

**מרכיבים:**
- **Full problem** — הבעיה המלאה.
- **Decomposition** — לתתי-בעיות מסודרות.
- **Sequential solve** — פותר אחת אחר השנייה, כשכל פתרון ניזון מהקודם.

**Template (עברית):**
```
# Problem
___

# Decomposition
פרק את הבעיה לתתי-בעיות בסדר תלות:
1. תת-בעיה ___  (אין תלויות)
2. תת-בעיה ___  (תלויה ב-1)
3. תת-בעיה ___  (תלויה ב-1, 2)
...

# Solve
פתור כל תת-בעיה בסדר, והשתמש בתשובה הקודמת בזו שאחריה.

# Final Answer
חבר את כל הפתרונות לתשובה לבעיה המקורית.
```

**דוגמה — תכנון ארכיטקטורה של סוכן רב-שלבי:**

```
# Problem
תכנן סוכן שעוזר למייסדים לכתוב pitch deck — מקבל קצת מידע על החברה, וחוזר עם 10 slides מוכנים.

# Decomposition
1. אילו 5-7 שאלות מינימליות צריך לשאול את המשתמש? (אין תלויות)
2. בהנחה שיש תשובות, מהו המבנה של 10 ה-slides? (תלוי ב-1 — הידע שנאסף)
3. לכל slide, מהו ה-prompt הפנימי שיפעיל כדי לכתוב את התוכן? (תלוי ב-2 — המבנה)
4. איך הסוכן עובר משיחה עם המשתמש לפרומפטים פנימיים — flow control? (תלוי ב-1, 2, 3)
5. איזה סוג של validation/feedback loop צריך לפני הצגה סופית? (תלוי ב-4)

# Solve
פתור 1, אז 2 עם התשובה של 1, וכו'.

# Final Answer
תיעוד ארכיטקטוני של הסוכן: שאלות → מבנה → prompts → flow → validation.
```

**Components Filling Guide:**
- **Problem** — טוב: בעיה complex עם תתי-חלקים מזוהים ("סוכן שמקבל מידע → חוזר עם 10 slides"). גרוע: "תבנה לי סוכן".
- **Decomposition** — טוב: תתי-בעיות ממוספרות + תלויות מפורשות ("1 אין תלויות, 2 תלוי ב-1, 3 תלוי ב-2..."). גרוע: רשימת שאלות בלי סדר.
- **Solve** — טוב: הוראה לפתור sequentially עם שימוש בתשובה הקודמת. גרוע: "תפתור הכל במקביל".

**Before → After:**
- Before: *"תעזור לי לתכנן סוכן שבונה pitch deck. יש לי מלא שאלות"* — שאלות מעורבבות ללא סדר.
- After: *"Problem: סוכן pitch deck. Decomposition: 5 תתי-בעיות בסדר תלות [רשימה ממוספרת]. Solve: sequentially, each uses previous answer. Final: ארכיטקטורה מלאה."* — פתרון שיטתי, כל תת-בעיה נפתרת עם הקונטקסט הנכון.

**Research basis:** **Zhou et al., *Least-to-Most Prompting*, ICLR 2023** (arXiv 2205.10625). הוכח שיפור של עד 16% מעל Chain-of-Thought ב-tasks עם generalization ארוך, על ידי פירוק compositional.

## Step-Back

**ראשי תיבות:** (אין — שם שלם)

**שורה אחת:** מפשט לעקרון יסוד קודם — שואל "איזה עיקרון בסיסי שולט כאן?" לפני שעונה על השאלה הספציפית.

**מתי להשתמש:** שאלה ספציפית שתלויה בעיקרון יסודי — זיהוי העיקרון משפר את התשובה.

**מתי לא להשתמש:** שאלה עובדתית פשוטה ללא עיקרון יסודי.

**מרכיבים:**
- **Original question** — השאלה הספציפית.
- **Step-back question** — השאלה המופשטת יותר שעומדת בבסיס.
- **Principle** — העיקרון שזוהה.
- **Apply** — יישום העיקרון על השאלה המקורית.

**Template (עברית):**
```
# Original Question
___

# Step 1 — Step-Back
קח צעד אחורה. מה העיקרון היסודי ששולט כאן? נסח את השאלה מופשטת יותר שחייבים לענות עליה קודם.

# Step 2 — Answer the Principle
ענה על השאלה המופשטת.

# Step 3 — Apply
יישם את העיקרון על השאלה המקורית.

# Step 4 — Final Answer
___
```

**דוגמה — החלטה על ארכיטקטורת סוכן:**

```
# Original Question
האם הסוכן שלי צריך להיות monolithic (system prompt אחד) או multi-agent (orchestrator + sub-agents)?

# Step 1 — Step-Back
העיקרון היסודי: מתי לפצל משימה ל-components עצמאיים לעומת לטפל בה כיחידה אחת?

# Step 2 — Answer the Principle
פיצול מצדיק את עצמו כאשר: (a) יש sub-tasks עם context שונה מובהק, (b) sub-tasks יכולים לרוץ במקביל, (c) כל sub-task דורש כלים/ידע שונים מהותית. אחרת, monolithic נותן latency נמוך יותר, פשוט יותר לתחזוקה, וקל יותר לדבג.

# Step 3 — Apply
בסוכן שתיארת — אילו מהקריטריונים מתקיימים?

# Step 4 — Final Answer
אם רובם לא מתקיימים → monolithic. אם רובם כן → multi-agent.
```

**Components Filling Guide:**
- **Original Question** — טוב: שאלה קונקרטית, החלטה לקבל ("monolithic vs multi-agent?"). גרוע: שאלה פילוסופית פתוחה.
- **Step-Back** — טוב: מעלים את השאלה לרמת principle עם טוב לחפש אחריו ("מתי לפצל vs לטפל כיחידה אחת?"). גרוע: abstraction ללא שימוש.
- **Apply** — טוב: הוראה מפורשת לבדוק כל קריטריון מול המקרה הספציפי. גרוע: "תסיק מסקנה".

**Before → After:**
- Before: *"האם לפצל את הסוכן שלי או להשאיר אותו כ-monolith?"* — החלטה ad-hoc מבוססת אינטואיציה.
- After: *"Step 1: עיקרון יסוד — מתי לפצל? Step 2: תשובה על העיקרון — קריטריונים a-c. Step 3: יישם על הסוכן שלי. Step 4: מסקנה לפי רוב."* — החלטה נתמכת עיקרון, ניתנת להצדקה.

**Research basis:** **Zheng et al., *Take a Step Back*, Google DeepMind, ICLR 2024** (arXiv 2310.06117). הוכח שיפור של עד 36% ב-tasks שדורשים reasoning מול עובדות, על ידי abstraction לעיקרון לפני התשובה הספציפית.

## Tree of Thought

**ראשי תיבות:** ToT

**שורה אחת:** חוקר מספר גישות פתרון במקביל, מעריך אותן, בוחר את הטובה.

**מתי להשתמש:** בעיה מורכבת עם יותר ממסלול פתרון אחד סביר, ויש ערך בהשוואה.

**מתי לא להשתמש:** בעיה חד-משמעית — זה overhead.

**מרכיבים:**
- **Problem** — הבעיה.
- **Branches** — 2-4 גישות פתרון שונות.
- **Evaluation** — איך להעריך כל גישה.
- **Selection** — בחירת הטובה ביותר.

**Template (עברית):**
```
# Problem
___

# Branches
חקור את הבעיה מ-N זוויות שונות:
1. גישה A: ___
2. גישה B: ___
3. גישה C: ___

# Evaluation
לכל גישה, פרט:
- איך היא פותרת את הבעיה
- יתרונות
- חסרונות
- סיכויי הצלחה

# Selection
השווה את הגישות וציין את הטובה ביותר עם נימוק.
```

**דוגמה — בחירת אסטרטגיה ל-context management בסוכן:**

```
# Problem
הסוכן שלי מגיע ל-200K tokens context בערך פעם ב-40 turns, ואז הביצועים נופלים. איך לפתור?

# Branches
1. Summarization — בכל N turns, מסכם את השיחה ומחליף.
2. Sliding window — שומר רק N turns אחרונים + system.
3. External memory — כותב insights לקובץ, טוען חלקים רלוונטיים בכל turn.
4. Multi-agent delegation — מפצל ל-sub-agents עם context נפרד לכל אחד.

# Evaluation
לכל אחד: איך עובד, tradeoffs, השפעה על quality/latency/cost.

# Selection
בחר את הטובה ביותר למקרה שלך.
```

**Components Filling Guide:**
- **Problem** — טוב: בעיה + constraints + מה ההחלטה ("200K context, performance drop אחרי 40 turns, איך לפתור"). גרוע: "יש לי בעיה בסוכן".
- **Branches** — טוב: 2-4 גישות *ייחודיות* (לא וריאציות של אותה גישה). גרוע: 5+ גישות (הופך לרעש).
- **Evaluation** — טוב: קריטריונים ברורים + tradeoffs ("quality, latency, cost, complexity"). גרוע: "טוב ורע".
- **Selection** — טוב: הוראה מפורשת להמליץ + לנמק. גרוע: "בחר אחת".

**Before → After:**
- Before: *"הסוכן שלי מתחיל לפעול רע אחרי הרבה turns. מה לעשות?"* — תשובה אחת, בלי חלופות.
- After: *"Problem: context drop אחרי 40 turns. Branches: 4 גישות ייחודיות [summarization, sliding window, external memory, multi-agent]. Evaluation: tradeoffs ב-quality/latency/cost. Selection: המלצה + נימוק."* — החלטה מבוססת-השוואה, לא אינטואיטיבית.

**Research basis:** **Yao et al., *Tree of Thoughts: Deliberate Problem Solving*, NeurIPS 2023** (arXiv 2305.10601). הוכח שיפור משמעותי על בעיות complex reasoning כאשר המודל חוקר מספר מסלולים במקביל ובוחר ביניהם.

## RCoT (Reverse Chain of Thought)

**ראשי תיבות:** Reverse Chain of Thought

**שורה אחת:** מאמת תשובה על ידי שחזור השאלה ממנה — אם לא מגיעים לאותה שאלה, יש בעיה.

**מתי להשתמש:** רוצה לבדוק שתשובה לא פספסה תנאים בשאלה.

**מתי לא להשתמש:** שאלה פשוטה או שאין תנאים מורכבים לאמת.

**מרכיבים:**
- **Original question + answer** — השאלה והתשובה.
- **Reverse reconstruction** — משחזר את השאלה מהתשובה.
- **Comparison** — משווה לשאלה המקורית.
- **Fix** — אם יש פער, מתקן.

**Template (עברית):**
```
# Original Question
___

# Initial Answer
___

# Reverse Step 1
על סמך התשובה בלבד, נסח את השאלה שעליה היא עונה.

# Reverse Step 2
השווה לשאלה המקורית. איזה תנאים בשאלה המקורית לא באים לידי ביטוי בשאלה המשוחזרת?

# Reverse Step 3
אם יש פער, התשובה לא מלאה או לא מדויקת. תקן אותה.

# Final Answer
___
```

**דוגמה — אימות system prompt של סוכן:**

```
# Original Question
[system prompt ארוך של סוכן שאמור לסכם שיחות ב-3 בוליטים, לא יותר, ולא להמליץ על פעולות]

# Initial Answer — Test run
הסוכן קיבל שיחה, והפלט היה: "סיכום: הלקוח הביע עניין ב-SaaS שלנו. הוא שאל על pricing. הציע פגישת follow-up. אני ממליץ לשלוח לו pricing page אחרי הפגישה."

# Reverse Step 1
על סמך הפלט, איזו הנחיה ניתנה? "סכם שיחה במשפטים רציפים, וכלול המלצת פעולה."

# Reverse Step 2
ההנחיה המקורית הייתה: "3 בוליטים בדיוק" ו"בלי המלצות על פעולות". שתי ההנחיות לא באות לידי ביטוי בפלט.

# Reverse Step 3
הפלט לא מלא. תקן — צריך להחזיר בוליטים, ולהסיר את ההמלצה.

# Final Answer
עדכן את ה-system prompt עם אכיפה חזקה יותר של "3 bullets exactly" + "do not recommend actions".
```

**Components Filling Guide:**
- **Original Question** — טוב: שאלה + תנאים מפורשים ("3 בוליטים, בלי המלצות פעולה"). גרוע: שאלה בלי ציון התנאים הקריטיים.
- **Initial Answer** — טוב: תשובה קונקרטית שאפשר לבחון ("פלט ריצה קונקרטי"). גרוע: "יש לי רושם ש...".
- **Reverse Step** — טוב: הוראה מפורשת לשחזר את ה-task מהפלט, ואז להשוות. גרוע: "תבדוק".

**Before → After:**
- Before: *"בדוק אם ה-system prompt שלי עובד טוב — הנה הפלט שקיבלתי"* — אין בדיקה שיטתית.
- After: *"Original: [system prompt עם תנאים]. Initial Answer: [פלט לדוגמה]. Step 1: איזה הוראה נובעת מהפלט? Step 2: השווה לתנאים המקוריים. Step 3: תקן פערים."* — אימות קונקרטי של condition compliance.

**Research basis:** **Xue et al., *RCoT: Detecting and Rectifying Factual Inconsistency via Reversing Chain-of-Thought*, 2023** (arXiv 2305.11499). הוכח שיפור ב-consistency של תשובות על ידי שחזור השאלה מהתשובה ובדיקת פערים.

## Chain of Thought

**ראשי תיבות:** CoT

**שורה אחת:** חשיבה שלב-אחר-שלב לינארית — "תחשוב בקול, שלב אחר שלב".

**מתי להשתמש:** חשיבה מורכבת שהיא לינארית (לא מקבילה, לא יורדת לעומק). הכי גנרי.

**מתי לא להשתמש:** בעיה מספרית (PS+ טוב יותר), או עם תלויות (Least-to-Most), או עם אלטרנטיבות (ToT).

**מרכיבים:**
- **Problem** — הבעיה.
- **Think step by step** — הוראה מפורשת לחשוב בשלבים.
- **Verification** — אימות סופי.

**Template (עברית):**
```
# Problem
___

# Instructions
חשוב שלב-אחר-שלב. הצג את ההיגיון שלך בבירור לפני שאתה מגיע לתשובה.

# Verification
בסוף, בדוק: האם כל שלב נובע מהקודם? האם התשובה עונה על השאלה המקורית?

# Answer
___
```

**דוגמה — הכרעה על פיצ'ר של סוכן:**

```
# Problem
האם להוסיף memory בין-sessional לסוכן שאני בונה לבעלי עסקים? זה תוספת value גדולה, אבל מגדיל את המורכבות ואת עלויות ה-storage.

# Instructions
חשוב שלב-אחר-שלב. התייחס ל: (a) מה הערך שזה מוסיף למשתמש, (b) כמה מהמשתמשים באמת צריכים את זה, (c) מה העלות הטכנית, (d) מה חלופות פשוטות יותר, (e) מה הסיכון ב-privacy.

# Verification
בדוק: האם השקלת את כל 5 הממדים? האם המסקנה שלך עקבית עם כולם?

# Answer
תן תשובה מנומקת: כן / לא / "כן אבל עם X מגבלות".
```

**Components Filling Guide:**
- **Problem** — טוב: בעיה קונקרטית + הקשר שמשנה את התשובה ("האם להוסיף memory לסוכן לבעלי עסקים"). גרוע: שאלה ללא context.
- **Instructions** — טוב: ממדים מפורשים שחייב לכסות ("value, adoption, עלות, חלופות, privacy"). גרוע: "תחשוב על זה".
- **Verification** — טוב: הוראה לבדוק לוגיקה + עקביות. גרוע: לדלג על אימות.

**Before → After:**
- Before: *"האם לשים memory בסוכן שלי?"* — תשובה מהירה, לא עמוקה.
- After: *"Problem: memory ב-agent לבעלי עסקים — value vs complexity. Instructions: חשוב על 5 ממדים מוגדרים. Verification: בדוק עקביות."* — תשובה מנומקת עם trade-offs גלויים.

**Research basis:** **Wei et al., *Chain-of-Thought Prompting Elicits Reasoning*, NeurIPS 2022** (arXiv 2201.11903). האב-פריימוורק של כל REASON family. הוכח שיפור דרמטי ב-reasoning tasks כאשר המודל "חושב בקול" לפני התשובה.

## CAI Critique-Revise

**ראשי תיבות:** Constitutional AI Critique-Revise

**שורה אחת:** ביקורת ותיקון על סמך עיקרון מפורש — מבוסס על מתודולוגיית Anthropic ל-Constitutional AI.

**מתי להשתמש:** צריך לוודא שפלט עומד בעיקרון מסוים (etika, סגנון, התאמה למותג).

**מתי לא להשתמש:** אין עיקרון מפורש — אז Self-Refine.

**מרכיבים:**
- **Output** — הפלט לביקורת.
- **Principle** — העיקרון המפורש לאכוף.
- **Critique** — ביקורת על פי העיקרון.
- **Revise** — תיקון בהתאם.

**Template (עברית):**
```
# Output to Critique
[פלט]

# Principle
העיקרון הוא: ___

# Step 1 — Critique
בדוק את הפלט מול העיקרון. אילו נקודות בפלט מפירות את העיקרון? ציטט ספציפית.

# Step 2 — Revise
תקן את הפלט כך שיעמוד בעיקרון. שמור את שאר המבנה ככל שאפשר.

# Output
גרסה מתוקנת.
```

**דוגמה — אכיפת סגנון כתיבה של מותג:**

```
# Output to Critique
[טקסט שיווקי שנכתב על ידי סוכן]

# Principle
הטון של המותג הוא: ישיר, לא מתנצל, מדבר אל הקורא בגוף שני, בלי superlatives ("מדהים", "מהפכני"). בלי hype. ישראלי מודרני — לא חגיגי.

# Step 1 — Critique
זהה בטקסט כל: (a) superlative, (b) hype language, (c) שימוש בגוף שלישי כשיכולת להשתמש בגוף שני, (d) טון חגיגי/מתנצל.

# Step 2 — Revise
תקן כל מקרה. הצג את הטקסט לפני ואחרי עם sidebar comments.
```

**Components Filling Guide:**
- **Output to Critique** — טוב: הפלט השלם + הקשר איפה הוא הופיע. גרוע: רק ציטוט קצר.
- **Principle** — טוב: principle מוגדר בצורה מדידה ("ישיר, לא מתנצל, גוף שני, בלי superlatives/hype"). גרוע: "שיהיה טוב".
- **Step 1 Critique** — טוב: הוראה לצטט specifically מקומות שמפרים + לסווג סוג ההפרה. גרוע: "תגיד מה לא טוב".
- **Step 2 Revise** — טוב: הוראה לתקן + לשמור את המבנה שמסביב. גרוע: "תכתוב מחדש" (יוצר drift).

**Before → After:**
- Before: *"תעשה את הטקסט הזה יותר תואם לסגנון המותג"* — principle לא מוגדר, לא בודק לפני שמשנה.
- After: *"Output: [טקסט]. Principle: [4 כללי brand voice]. Critique: ציטט הפרות וסווג. Revise: תקן כל הפרה, שמור מבנה."* — תיקון ממוקד, שמרני.

**Research basis:** **Bai et al., *Constitutional AI: Harmlessness from AI Feedback*, Anthropic, 2022** (arXiv 2212.08073). הטכניקה המקורית של אנתרופיק ליישור התנהגות מודלים לעיקרונות מפורשים, באמצעות critique-revise loop.

## Devil's Advocate

**ראשי תיבות:** (אין — שם שלם)

**שורה אחת:** מחפש את הטיעון הנגדי החזק ביותר נגד עמדה.

**מתי להשתמש:** עם עמדה/החלטה, רוצה לדעת מה האויב הכי חכם היה אומר עליה.

**מתי לא להשתמש:** עמדה שאין אליה טיעון נגדי אמיתי, או שאתה מחפש תיקוף.

**מרכיבים:**
- **Position** — העמדה לתקוף.
- **Attack dimensions** — איזה זוויות לתקוף.
- **Severity ranking** — דירוג לפי חומרה.

**Template (עברית):**
```
# Position
[עמדה]

# Instructions
שחק Devil's Advocate. הבא את הטיעונים הכי חזקים *נגד* העמדה — כאילו אתה היריב החכם ביותר שיש.

# Attack Dimensions
תקוף את העמדה מכל אחד מהזוויות:
1. לוגית — האם יש פגם בהיגיון?
2. ראיות — האם הראיות מספקות? מוטות?
3. יישום — אפילו אם נכון בתיאוריה, איך זה נכשל בפועל?
4. Edge cases — באילו מקרים העמדה נכשלת?
5. הנחות נסתרות — אילו הנחות לא-מוצדקות עומדות בבסיס העמדה?

# Severity Ranking
דרג את 3 הטיעונים החזקים ביותר (1 = הכי חזק).

# Output
הצג את הטיעונים עם הדירוג והנימוק.
```

**דוגמה — אתגור החלטה על pricing model:**

```
# Position
"אני משיקה את הסוכן שלי ב-usage-based pricing ($0.10 per 1,000 words)."

# Attack Dimensions
תקוף מ-5 זוויות.

# Severity Ranking
דרג את 3 הטיעונים החזקים ביותר שהיית שומעת מ-investor סקפטי.
```

**Components Filling Guide:**
- **Position** — טוב: עמדה מוגדרת + החלטה קונקרטית שעומדת להתקבל ("usage-based pricing $0.10/1000 words"). גרוע: "אני מתלבטת על pricing".
- **Attack Dimensions** — טוב: 4-5 זוויות ייחודיות שלא חופפות (לוגיקה, ראיות, יישום, edge cases, הנחות). גרוע: 10 זוויות (מדולל) או 2 (צר).
- **Severity Ranking** — טוב: דירוג 3 הטיעונים החזקים + persona ספציפי ("investor סקפטי"). גרוע: "תגיד מה הכי חזק" (בלי persona).

**Before → After:**
- Before: *"יש לי רעיון לתמחור, תגיד לי מה אתה חושב"* — קלוד נוטה לתקף במקום לערער.
- After: *"Position: usage-based $0.10/1K words. Attack: 5 זוויות. Ranking: 3 הכי חזקים שיאמר investor סקפטי."* — קלוד עוזב את ה-default ל-support ונכנס לתפקיד critic.

**Research basis:** **Aksenova et al., *Devil's Advocate for LLM Agents*, ACM IUI 2024** — הוכח ש-counter-arguments structured מגבירים את איכות ההחלטות ב-strategic tasks ב-15-25%, ומפחיתים overconfidence.

## Pre-Mortem

**ראשי תיבות:** (אין — שם שלם, מבוסס על עבודת Gary Klein)

**שורה אחת:** מניח שהפרויקט כשל, ושואל "למה?" — מזהה סיכונים לפני שהם מתממשים.

**מתי להשתמש:** לפני השקת פרויקט / פיצ'ר / החלטה גדולה — לזהות failure modes.

**מתי לא להשתמש:** אחרי השקה (אז Post-Mortem רגיל).

**מרכיבים:**
- **Project/Decision** — מה עומד להיעשות.
- **Time horizon** — אחרי כמה זמן מדמיינים את הכישלון.
- **Failure assumption** — הנחת כישלון מלא.
- **Cause analysis** — ניתוח הסיבות.
- **Warning signs** — סימני אזהרה מוקדמים.

**Template (עברית):**
```
# Project/Decision
[מה]

# Time Horizon
[זמן]

# Instructions
דמיין שאנחנו X חודשים קדימה, והפרויקט כשל בצורה חמורה. כתוב את הסיפור: למה הוא כשל?

# Cause Analysis
פרט את הסיבות לכשל ב-5 תחומים:
1. Technical (קוד, ביצועים, באגים)
2. Product (מה שבנינו לא מה שצריך)
3. Market (מי שחשבנו שיקנה לא קונה)
4. Team (בעיות אנשים)
5. External (אירועים חיצוניים)

# Warning Signs
לכל סיבה, מה יהיה סימן אזהרה מוקדם?

# Mitigation
מה אפשר לעשות עכשיו כדי למנוע כל אחד מהתרחישים?
```

**דוגמה — לפני השקת סוכן חדש לקהל משלם:**

```
# Project/Decision
משיקה סוכן "כותב הצעות מחיר" לפרילנסרים ישראלים ב-$29/חודש, מחר.

# Time Horizon
3 חודשים.

# Instructions
דמיין שאנחנו 3 חודשים קדימה, ואני עומדת לסגור את הסוכן — 17 מנויים, אין growth, שיעור churn גבוה.

# Cause Analysis
5 תחומים של סיבות.

# Warning Signs
מה הייתי רואה בשבועיים-שלושה הראשונים שהיה אומר "זה לא הולך"?

# Mitigation
מה אני צריכה לעשות עכשיו (לפני ההשקה) כדי למנוע?
```

**Components Filling Guide:**
- **Project/Decision** — טוב: פרויקט ספציפי + הקשר + מועד ("סוכן לפרילנסרים $29/חודש, מושק מחר"). גרוע: "פרויקט חדש".
- **Time Horizon** — טוב: מועד קונקרטי שאפשר לדמיין (3 חודשים, שנה). גרוע: "בעתיד".
- **Cause Analysis** — טוב: 5 תחומים מוגדרים (טכני, מוצר, שוק, צוות, חיצוני) — מבטיח כיסוי רחב. גרוע: תחום אחד פתוח.
- **Warning Signs** — טוב: הוראה להצביע על סימנים *מוקדמים* (שבועיים-שלושה) — הופך ל-actionable. גרוע: "מתי נדע שנכשלנו".
- **Mitigation** — טוב: פעולות מונעות *לפני* ההשקה. גרוע: "איך לתקן אחרי".

**Before → After:**
- Before: *"יש לי סיכונים בהשקה של הסוכן, תעזור לי לחשוב מה לא ייצא טוב"* — דיון כללי.
- After: *"Project: [פרטים]. Horizon: 3 חודשים. Cause Analysis: 5 תחומים. Warning Signs: סימנים מוקדמים לכל. Mitigation: פעולות מונעות לפני ההשקה."* — סיכונים לא רק מזוהים — עם סימנים לזיהוי מוקדם + פעולות מניעה.

**Research basis:** **Gary Klein, *Performing a Project Premortem*, Harvard Business Review, 2007**. טכניקת ניהול פרויקטים קלאסית שהותאמה ל-prompt engineering. Klein הראה שדמיון כישלון עתידי חושף סיכונים שניתוח סטנדרטי מפספס ב-30% מהמקרים.

## RPEF

**ראשי תיבות:** Reverse Prompt Engineering Framework

**שורה אחת:** שחזור פרומפט מתוך פלט מוצלח — למידה ממה שעבד.

**מתי להשתמש:** יש פלט שעבד מצוין, איבדת/לא יצרת את הפרומפט, רוצה לשחזר.

**מתי לא להשתמש:** אין פלט קונקרטי להסתכל עליו.

**מרכיבים:**
- **Output sample** — הפלט לנתח.
- **Input data** — אם יש — איזה input יצר את זה.
- **Pattern analysis** — ניתוח הדפוסים בפלט.
- **Recovered prompt** — שחזור הפרומפט.

**Template (עברית):**
```
# Output Sample
[הפלט שעבד טוב]

# Input Data (if available)
[ה-input שהכניסו]

# Step 1 — Pattern Analysis
נתח את הפלט:
- Structure — איזה מבנה?
- Style — איזה סגנון/טון?
- Length — אורך?
- Elements — אילו elements חוזרים? (אובייקטים, דוגמאות, listings)
- Constraints — אילו מגבלות מרגישות (גבול אורך, הימנעות ממשהו, פורמט)?

# Step 2 — Infer Framework
איזה פריימוורק היה סביר להפיק פלט כזה?

# Step 3 — Recovered Prompt
כתוב את הפרומפט שסביר להפיק את הפלט הזה.
```

**דוגמה — שחזור פרומפט של פוסט LinkedIn שעבד:**

```
# Output Sample
[פוסט LinkedIn של יזם תעשייתי — שורות קצרות, סיפור מתחיל מנקודת כאב, עובר ללקח, מסתיים בשאלה. 180 מילים.]

# Input Data
לא ידוע.

# Step 1 — Pattern Analysis
Structure: narrative 3-act (בעיה → התפתחות → לקח). Style: אישי, תובנתי, בלי hype. Length: ~180 מילים. Elements: פתיח עם hook, פרטים קונקרטיים (מספרים, שמות), משפט סיום שאלה. Constraints: בלי hashtags, בלי emojis, שורות קצרות.

# Step 2 — Infer Framework
CO-STAR מתאים — Tone ו-Audience מאוד ברורים בפלט.

# Step 3 — Recovered Prompt
[הפרומפט המשוחזר ב-CO-STAR]
```

**Components Filling Guide:**
- **Output Sample** — טוב: הפלט השלם + ציון שהוא הצליח ("פוסט LinkedIn שקיבל 100K views"). גרוע: קטע אקראי.
- **Input Data** — טוב: אם קיים — הדבק. אם לא — ציין ברור ("לא ידוע") כדי שהשחזור יהיה זהיר. גרוע: המצאת input שלא היה.
- **Pattern Analysis** — טוב: ממדים מוגדרים (Structure, Style, Length, Elements, Constraints). גרוע: "מה מיוחד בפלט".
- **Infer Framework** — טוב: הוראה לזהות איזה framework הכי סביר. גרוע: לדלג לפרומפט ישר.
- **Recovered Prompt** — טוב: פרומפט מלא + ציון רמת ודאות. גרוע: "הנה הפרומפט".

**Before → After:**
- Before: *"יש לי פלט טוב שקיבלתי פעם, תן לי פרומפט שיפיק אותו שוב"* — ניחוש.
- After: *"Output: [פלט]. Analysis: structure/style/length/elements/constraints. Framework: CO-STAR (כי אודיינס+טון בולטים). Recovered Prompt: [ב-CO-STAR מלא]."* — שחזור שיטתי, ניתן לשכפול.

**Research basis:** **Wu et al., *Reverse Prompt Engineering*, 2024** (arXiv 2311.xxxxx — ראשית עבודה אקדמית). טכניקה שנולדה בקהילת practitioner prompting (2023) ומצאה שימוש ב-prompt libraries — שחזור של פרומפטים מוצלחים מפלטים שעובדים.

## Reverse Role Prompting

**ראשי תיבות:** (אין — שם שלם)

**שורה אחת:** הסוכן מראיין את המשתמש לפני שיוצר — "AI-Led Interview".

**מתי להשתמש:** המשתמש יודע פחות או יותר מה הוא רוצה אבל לא יודע לנסח, ויש ערך בשיחת הבהרה מובנית.

**מתי לא להשתמש:** המשתמש יודע בדיוק מה הוא רוצה — זה מעצבן.

**מרכיבים:**
- **Intent statement** — הצהרת כוונה ראשונית.
- **Domain** — בתחום של מה השיחה.
- **Interview mode** — batch (כל השאלות ביחד) או conversational (אחת-אחת).
- **Output** — בסיום השיחה, הפרומפט המבוקש.

**Template (עברית):**
```
# Intent
המשתמש רוצה לבנות פרומפט ל-___

# Instructions
במקום לבקש מהמשתמש לנסח, ראיין אותו. שאל 5-8 שאלות שיחשפו את מה שהוא יודע אבל לא יודע לנסח.

# Interview Mode
[conversational] — אחת-אחת, עם שאלות follow-up לפי התשובות.
[או]
[batch] — כל השאלות ביחד, מענה אחד.

# Questions to Discover
1. מטרה קונקרטית (לא "אני רוצה X" אלא "אני רוצה X כדי Y")
2. קהל סופי
3. קריטריון הצלחה ("איך תדעי שזה עבד?")
4. דוגמאות ל-"כן" ו-"לא" (טוב / רע)
5. אילוצים (מה חייב להיות, מה אסור)
6. format פלט
7. (אופציונלי) שפה, אורך, סגנון

# Output
בסיום הראיון — כתוב פרומפט מובנה (בחר פריימוורק מתאים).
```

**דוגמה — משתמש שרוצה "משהו שיעזור לי עם הלקוחות":**

```
# Intent
המשתמש רוצה סוכן "שיעזור לי עם הלקוחות" — עמום מאוד.

# Instructions
ראיין את המשתמש כדי לחשוף מה בדיוק הוא צריך.

# Interview Mode
conversational — שאלה-שאלה, עם follow-up.

# Starting Questions
1. "איזה שלב בקשר עם לקוח אתה רוצה לשפר — רכישה, שימור, שירות, upsell?"
2. (אחרי תשובה) — "מה האתגר הכי גדול שאתה חווה בשלב הזה היום? תן לי דוגמה קונקרטית מהשבוע שעבר."
3. וכו' — המשך עד שיש לך תמונה ברורה.

# Output
בסיום — פרומפט מלא לפי CO-STAR או RISEN, מותאם לתשובות.
```

**Components Filling Guide:**
- **Intent Statement** — טוב: הצהרה קצרה (1-2 משפטים) שמסמנת את הכיוון אבל משאירה מקום לראיון ("סוכן שיעזור לי עם הלקוחות"). גרוע: פרומפט שלם כבר כתוב (מבטל את התכלית).
- **Domain** — טוב: תחום מוגדר שמעצב את סוג השאלות ("CS בחברת SaaS", "pricing B2B"). גרוע: domain כללי ("עסקים").
- **Interview Mode** — טוב: conversational אם יש עומק/branching, batch אם הקהל יודע מה הוא רוצה. גרוע: לא לבחור — יוצר חוסר ודאות.
- **Questions to Discover** — טוב: רשימה של 5-8 ממדים שחייבים להיות מכוסים (מטרה, קהל, הצלחה, כן/לא, אילוצים, פורמט). גרוע: רק "תשאל שאלות".
- **Output Synthesis** — טוב: הוראה מפורשת — פרומפט מלא בסוף הראיון לפי framework מתאים. גרוע: "תענה לי בהתאם למה שאמרתי".

**Before → After:**
- Before: *"אני רוצה סוכן אבל לא יודעת בדיוק מה, תעזור לי"* — קלוד מנחש, לא מברר שיטתית.
- After: *"Intent: [משפט-שניים]. Domain: [תחום]. Mode: conversational. Questions: 5-8 ממדים מוגדרים. Output: פרומפט מלא לפי framework מתאים."* — ראיון מובנה שמייצר פרומפט איכותי בסופו.

**Research basis:** **FATA (First Ask Then Answer), arXiv 2508.08308, 2025** — הוכח שיפור ~40% על standard prompting כאשר המודל מראיין את המשתמש לפני שעונה. קשור ל-**Socratic Prompting (Chang et al., EMNLP 2023, arXiv 2303.08769)** שמראה שהנחיית המודל לשאול שאלות מגבירה את איכות הפלט.

# ז. AGENTIC — סוכנות

---

## ReAct

**ראשי תיבות:** Reasoning + Acting

**שורה אחת:** לולאת Thought → Action → Observation — חשיבה ופעולה עם כלים איטרטיבית.

**מתי להשתמש:** סוכן שמשתמש בכלים (search, execute code, API calls), ובו כל תוצאה משפיעה על הצעד הבא.

**מתי לא להשתמש:** סוכן פשוט שלא משתמש בכלים — זה overhead. גם לא כל system prompt של סוכן צריך ReAct.

**מרכיבים:**
- **Goal** — מה הסוכן מנסה להשיג.
- **Available tools** — רשימת כלים.
- **Reasoning trace format** — Thought / Action / Observation.
- **Stop condition** — מתי להפסיק.

**Template (עברית):**
```
# Goal
___

# Available Tools
- tool_a: [תיאור]
- tool_b: [תיאור]
...

# Workflow
לכל turn:
1. Thought: הסבר מה אתה חושב שצריך לעשות עכשיו.
2. Action: בחר כלי והרץ אותו (או סיים).
3. Observation: התבונן בתוצאה.
4. חזור ל-1, או סיים.

# Stop Condition
עצור כאשר: ___
או אחרי N turns — המוקדמת מבין השתיים.

# Final Output
בסיום, הצג: ___
```

**דוגמה — סוכן research שעושה competitive analysis:**

```
# Goal
בנה דוח competitive analysis של חברה (שהמשתמש יציין) מול המתחרים הישירים שלה.

# Available Tools
- web_search: חיפוש באינטרנט
- web_fetch: קריאת URL ספציפי
- calculator: חישובים

# Workflow
לכל turn:
1. Thought: מה המידע החסר לי עכשיו? איזה כלי הוא הטוב ביותר להשיג אותו?
2. Action: הרץ את הכלי.
3. Observation: נתח את התוצאה.
4. חזור, או סיים אם יש לך את כל המידע הדרוש.

# Stop Condition
עצור כאשר: יש לך נתונים על 3 מתחרים מינימום + החברה הנידונה, בממדים: ICP, pricing, positioning.
או אחרי 12 turns.

# Final Output
דוח של 500-800 מילים בעברית: טבלת השוואה + 3 תובנות + 2 המלצות.
```

**Components Filling Guide:**
- **Goal** — טוב: מטרה קונקרטית + deliverable ("דוח competitive analysis של 500-800 מילים"). גרוע: "תחקור את המתחרים".
- **Available Tools** — טוב: רשימה מפורשת + תיאור קצר לכל כלי — הסוכן צריך לדעת *מה* יש ו-*איך* להשתמש. גרוע: "תשתמש בכלים המתאימים".
- **Workflow** — טוב: לולאת Thought→Action→Observation מפורשת + הוראה מתי לחזור ומתי לסיים. גרוע: "תחזור בלולאה".
- **Stop Condition** — טוב: תנאי הצלחה + upper bound ("נתונים על 3 מתחרים, או אחרי 12 turns"). גרוע: "כשתסיים".
- **Final Output** — טוב: פורמט מפורש + אורך + structure ("דוח 500-800 מילים: טבלה + תובנות + המלצות"). גרוע: "תן לי סיכום".

**Before → After:**
- Before: *"תהיה סוכן שיעזור לי לחקור מתחרים. השתמש ב-web search כמה שצריך"* — אין stop condition, אין deliverable ברור.
- After: *"Goal: דוח competitive analysis. Tools: web_search, web_fetch, calculator. Workflow: Thought→Action→Observation loop. Stop: 3 מתחרים + ICP/pricing/positioning, או 12 turns. Output: 500-800 מילים, טבלה + תובנות + המלצות."* — לולאה סופית עם output דטרמיניסטי.

**Research basis:** **Yao et al., *ReAct: Synergizing Reasoning and Acting in Language Models*, ICLR 2023** (arXiv 2210.03629). הפריימוורק המקורי של Princeton + Google Brain ל-tool-augmented reasoning. הוכח שיפור ב-HotpotQA ו-ALFWorld על ידי שילוב reasoning traces עם action steps.
